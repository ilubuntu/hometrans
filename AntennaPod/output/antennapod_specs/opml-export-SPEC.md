# SPEC: REQ-032 OPML导出

## 需求概述

用户可以将已订阅的播客列表导出为标准 OPML 2.0 格式文件，方便迁移到其他播客应用或备份。导出文件包含所有已订阅播客的标题、Feed URL、网页链接和类型信息。用户通过系统文件选择器选择导出路径。

## 原子场景

### 场景 32.1：导出全部订阅为 OPML（Happy Path）
**触发**: 用户在设置中点击"OPML 导出"
**执行步骤**:
1. 用户进入设置 → 存储 → OPML 导出
2. 应用打开系统文件选择器，让用户选择保存位置和文件名
3. 用户选择目标路径并确认
4. 应用获取所有已订阅播客列表
5. 生成标准 OPML 2.0 格式 XML 文档
6. 文档头部包含标题和创建日期
7. 每个已订阅播客以 outline 元素写入
8. 写入完成后提示用户导出成功

### 场景 32.2：导出文件格式正确性（Happy Path）
**触发**: 验证导出的 OPML 文件格式
**执行步骤**:
1. 导出文件为标准 XML 格式，UTF-8 编码
2. 根元素为 `<opml version="2.0">`
3. `<head>` 包含标题和 RFC822 格式日期
4. `<body>` 包含每个播客的 `<outline>` 元素
5. 每个 outline 包含：text、title、type、xmlUrl、htmlUrl 属性
6. 文件有缩进，便于阅读

### 场景 32.3：仅导出已订阅的播客（边界场景）
**触发**: 订阅列表中包含已退订的播客
**执行步骤**:
1. 应用遍历所有 Feed 数据
2. 仅导出状态为"已订阅"的播客
3. 已退订或未完成的播客不包含在导出文件中

### 场景 32.4：无订阅时导出（边界场景）
**触发**: 用户没有任何订阅时执行导出
**执行步骤**:
1. 应用获取订阅列表为空
2. 仍然生成有效的 OPML 文件
3. `<body>` 部分为空
4. 文件结构完整，可正常打开

### 场景 32.5：播客缺少网页链接（边界场景）
**触发**: 某些播客没有网页链接（htmlUrl）
**执行步骤**:
1. 导出时检测每个 Feed 的 link 字段
2. 有 link 的播客写入 htmlUrl 属性
3. 无 link 的播客省略 htmlUrl 属性
4. xmlUrl 和 title 始终写入

### 场景 32.6：取消导出（边界场景）
**触发**: 用户在文件选择器中取消操作
**执行步骤**:
1. 用户在系统文件选择器中按返回键
2. 不执行任何导出操作
3. 返回设置页面

### 场景 32.7：导出路径不可写（异常场景）
**触发**: 用户选择的目标路径不可写入
**执行步骤**:
1. 应用尝试写入文件
2. 写入失败，抛出 IO 异常
3. 显示错误提示信息
4. 用户可重新选择路径

## 验收要点
- 可通过系统文件选择器选择导出路径
- 导出标准 OPML 2.0 格式文件
- 仅导出已订阅状态的播客
- 每个 outline 包含标题、Feed URL，有网页链接时包含 htmlUrl
- 文件使用 UTF-8 编码，有缩进
- 文件头部包含标题和创建日期
- 无订阅时可生成有效的空 OPML 文件
