# SPEC: REQ-040 数据持久化

## 需求概述

应用的所有用户数据在应用重启后必须完整保持，包括：订阅的播客列表、播放队列（含排序顺序）、收藏列表、播放历史、已下载的文件记录、各播客的偏好设置、全局用户设置（主题/播放/下载/同步）、播放位置和进度。数据存储在本地数据库中，支持完整备份导出和恢复。系统级自动备份确保数据安全。

## 原子场景

### 场景 40.1：应用重启后恢复订阅列表（Happy Path）
**触发**: 用户关闭并重新打开应用
**执行步骤**:
1. 用户上次添加了多个播客订阅
2. 用户完全关闭应用后重新打开
3. 应用从本地数据库加载所有订阅数据
4. 订阅列表完整显示所有已订阅播客
5. 每个播客的封面、标题、描述完整恢复
6. 每个播客的单集列表完整恢复

### 场景 40.2：应用重启后恢复播放队列（Happy Path）
**触发**: 重启后恢复播放队列
**执行步骤**:
1. 用户上次排好了播放队列顺序
2. 重启应用
3. 应用从数据库加载队列数据
4. 播放队列完整恢复，排序顺序与上次一致
5. 队列中的每集状态（已播/未播）保持

### 场景 40.3：应用重启后恢复播放位置（Happy Path）
**触发**: 重启后继续播放
**执行步骤**:
1. 用户上次播放到某单集的 15:30 位置
2. 播放服务定期保存播放位置到数据库
3. 重启应用
4. 应用从数据库加载该单集的播放位置
5. 用户继续播放时从 15:30 位置恢复
6. 播放进度准确

### 场景 40.4：应用重启后恢复收藏列表（Happy Path）
**触发**: 重启后查看收藏
**执行步骤**:
1. 用户之前收藏了多集单集
2. 重启应用
3. 应用从数据库加载收藏数据
4. 收藏列表完整显示所有已收藏的单集

### 场景 40.5：应用重启后恢复播放历史（Happy Path）
**触发**: 重启后查看播放历史
**执行步骤**:
1. 用户之前播放过多集单集
2. 播放历史记录在数据库中
3. 重启应用
4. 播放历史完整恢复
5. 历史记录按时间排序

### 场景 40.6：应用重启后恢复已下载文件记录（Happy Path）
**触发**: 重启后查看已下载列表
**执行步骤**:
1. 用户之前下载了多集单集
2. 下载记录和文件路径存储在数据库中
3. 重启应用
4. 已下载列表完整恢复
5. 每集显示文件大小和下载状态
6. 可正常播放已下载的文件

### 场景 40.7：应用重启后恢复播客设置（Happy Path）
**触发**: 重启后检查各播客偏好设置
**执行步骤**:
1. 用户之前为各播客设置了不同偏好（自动下载、播放速度、标签等）
2. 播客偏好存储在数据库中
3. 重启应用
4. 每个播客的偏好设置完整恢复
5. 自动下载规则、播放速度、标签等保持不变

### 场景 40.8：应用重启后恢复全局设置（Happy Path）
**触发**: 重启后检查全局设置
**执行步骤**:
1. 用户之前设置了主题、播放偏好、下载规则、同步配置
2. 全局设置通过偏好存储持久化
3. 重启应用
4. 主题模式恢复（如深色模式）
5. 播放设置恢复（快进/快退秒数、入队位置）
6. 下载设置恢复（自动下载开关、更新间隔）
7. 同步凭证恢复（如已登录同步服务）

### 场景 40.9：数据库完整备份导出（Happy Path）
**触发**: 用户导出数据库备份
**执行步骤**:
1. 用户在设置中点击"数据库导出"
2. 选择导出目标路径
3. 应用导出完整数据库到文件
4. 包含所有表数据（订阅、队列、收藏、历史、设置）
5. 导出完成后提示用户

### 场景 40.10：数据库恢复导入（Happy Path）
**触发**: 用户从备份文件恢复数据库
**执行步骤**:
1. 用户在设置中点击"数据库导入"
2. 显示警告提示（将覆盖现有数据）
3. 用户确认后选择备份文件
4. 应用导入备份文件，覆盖当前数据库
5. 重启应用后所有数据恢复到备份时的状态

### 场景 40.11：系统自动备份（边界场景）
**触发**: 系统执行应用数据自动备份
**执行步骤**:
1. 系统备份服务触发应用备份
2. 应用将订阅列表导出为 OPML 格式
3. OPML 文件包含在系统备份中
4. 系统恢复时还原 OPML 备份

### 场景 40.12：定期自动数据库导出（边界场景）
**触发**: 用户开启了自动数据库导出
**执行步骤**:
1. 应用定期执行数据库导出任务
2. 导出到用户指定的自动备份目录
3. 文件名包含日期戳
4. 即使应用未打开也会在后台执行

### 场景 40.13：数据库升级数据迁移（边界场景）
**触发**: 应用更新后数据库结构变化
**执行步骤**:
1. 检测到数据库版本号变化
2. 执行数据库升级脚本
3. 迁移旧数据到新表结构
4. 保持用户数据不丢失
5. 必要时执行数据格式转换

### 场景 40.14：数据库损坏恢复（异常场景）
**触发**: 数据库文件损坏
**执行步骤**:
1. 应用打开数据库时检测到损坏
2. 数据库错误处理器介入
3. 尝试从备份恢复
4. 若无备份，重建空数据库
5. 应用不崩溃，可正常使用

## 验收要点
- 应用重启后所有数据完整恢复（订阅、队列、收藏、历史、下载、设置）
- 播放位置准确恢复到上次播放处
- 播放队列排序顺序保持
- 各播客偏好设置和全局设置完整恢复
- 同步凭证持久化
- 支持完整数据库导出和导入恢复
- 支持系统级自动备份（OPML）
- 支持定期自动数据库导出
- 数据库升级时自动迁移数据
- 数据库损坏时不崩溃，可恢复
