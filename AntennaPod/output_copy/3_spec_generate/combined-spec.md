# 首次启动空库首页SPEC

应用首次启动进入主界面，并在订阅库为空时展示 Home 页面的空状态与基础导航入口

## 场景一

### 场景概述

用户在设备桌面点击应用图标，应用首次启动并进入主界面的 Home 页面，同时初始化订阅源自动更新

### 场景逻辑步骤

- 1. 用户在系统桌面点击应用图标，系统启动应用的闪屏页面
- 2. 闪屏页面随即进入主界面，主界面默认加载 Home 页面作为初始页面
- 3. 若判定为首次启动（首次启动标记默认为开启），则启动订阅源周期性自动更新任务，并将首次启动标记写入配置存储置为关闭，后续启动不再重复初始化该任务
- 4. Home 页面顶部工具栏固定显示标题"Home"，并提供搜索入口与溢出菜单（刷新订阅源、配置首页）

## 场景二

### 场景概述

用户首次启动进入 Home 页面时订阅库为空（没有任何单集），页面展示空状态欢迎区

### 场景逻辑步骤

- 1. 应用进入 Home 页面后，异步查询数据库中的单集总数（不区分任何过滤条件）
- 2. 当单集总数为 0 时，Home 页面展示空状态欢迎区，内容包括：应用图标、欢迎标题"Welcome to AntennaPod!"、引导文案"You are not subscribed to any podcasts yet. Open the menu to add a podcast."
- 3. 同时隐藏 Home 页面的分区内容列表与下拉刷新区域，并将可滚动区域重置到顶部
- 4. 空状态欢迎区显示一个弯曲箭头，指向导航入口：若用户启用了底部导航，箭头指向底部导航栏；若使用侧边栏模式，箭头指向侧边栏方向

## 场景三

### 场景概述

空库状态下 Home 页面底部导航栏的展示与"More"溢出入口的性质

### 场景逻辑步骤

- 1. Home 页面底部展示导航栏，按用户当前可见的导航项顺序渲染最多 4 个主页面入口，并固定追加一个"More"入口
- 2. 默认顺序下，前 4 个主页面入口依次为：Home、Queue、Inbox、Subscriptions
- 3. "More"入口为溢出菜单入口，不是独立的内容页面；点击"More"时不会离开当前页面，而是在当前页面之上弹出菜单列表
- 4. Inbox 入口根据新单集数量显示角标（有新单集时显示数量，无则隐藏）

## 场景四

### 场景概述

空状态展示期间，当订阅源列表发生变化时，Home 页面重新判定并切换空状态

### 场景逻辑步骤

- 1. 当订阅源列表发生变更（例如完成一次订阅、退订或订阅源刷新完成）时，应用发出订阅源列表变更事件
- 2. Home 页面接收到该事件后，重新异步查询数据库中的单集总数
- 3. 若单集总数仍为 0，保持空状态欢迎区显示
- 4. 若单集总数大于 0，则隐藏欢迎区，显示 Home 页面的分区内容列表与下拉刷新区域

## 场景五

### 场景概述

"首次启动空库首页"功能的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下入口: 系统桌面的应用图标启动链路（闪屏页面 → 主界面 → Home 页面）；Home 页面顶部工具栏的搜索、刷新、配置首页入口；底部导航栏的 4 个主页面入口与"More"溢出入口
- 2. **不作用于** 以下位置: Queue、Inbox、Subscriptions 等其他主页面的内部内容（这些页面的空态由各自页面单独处理，不由本空状态逻辑承载）；播放页、单集详情页、播客详情页等次级页面不参与空状态切换
- 3. 空状态判定只读取数据库单集总数，不读取订阅列表本身：即"有订阅但订阅源尚无任何单集"时，仍判定为空状态

> [偏差] 需求描述空状态应"引导用户添加播客或管理订阅"，但实际空状态文案仅为"打开菜单添加播客"，未明确提及"管理订阅"入口；且空状态欢迎区内没有内联的"添加播客"或"Subscriptions"可点按钮，用户需通过底部导航的"More"菜单或导航项进入对应页面。

<!-- REQ-002: 底部导航基础结构 -->
# 底部导航基础结构SPEC

应用底部导航栏提供主页面切换与溢出菜单入口，主页面之间相互独立切换，More 为弹出菜单而非独立内容页

## 场景一

### 场景概述

用户在底部导航栏点击 Home、Queue、Inbox 或 Subscriptions 主页面入口，切换到对应主页面

### 场景逻辑步骤

- 1. 应用底部导航栏按用户当前可见的导航项顺序，渲染最多 4 个主页面入口与 1 个"More"入口；默认顺序下 4 个主页面依次为 Home、Queue、Inbox、Subscriptions
- 2. 用户点击其中某一个主页面入口（如 Queue）
- 3. 底部导航高亮被点击的入口，主内容区切换为对应主页面，并清空页面返回栈
- 4. 主页面自行渲染各自的顶部标题（如"Queue"），应用记录该页面为"当前主页面"，以便下次恢复时回到此处
- 5. 若被点击的入口不在当前底部导航可见的 4 项中（例如用户调整过导航组成），则高亮自动回退到"More"项，主内容区仍切换到目标页面

## 场景二

### 场景概述

用户点击底部导航的"More"入口，应用在当前页面之上弹出溢出菜单，不进入独立的"More"页面

### 场景逻辑步骤

- 1. 用户点击底部导航最右侧的"More"入口
- 2. 应用不切换主内容区，被点击的"More"入口也不产生高亮选中态（当前主页面高亮保持不变）
- 3. 在底部导航栏上方、靠右侧位置弹出一个浮层菜单列表，菜单覆盖在当前页面之上，当前页面内容不被替换
- 4. 菜单列表包含未在底部导航直接展示的其他主页面入口，以及"Customize navigation"（自定义导航）与"Settings"（设置）两个固定项
- 5. 用户点击菜单外区域或返回键可关闭该浮层菜单，关闭后回到原当前页面

## 场景三

### 场景概述

应用恢复或重新启动时，底部导航根据上次记录的当前主页面恢复高亮与内容

### 场景逻辑步骤

- 1. 应用启动进入主界面时，读取配置存储中记录的"当前主页面"
- 2. 若用户设置默认页为"记忆上次页面"，则加载上次记录的主页面；否则加载用户指定的默认页
- 3. 底部导航高亮该主页面对应的入口；若该入口不在当前可见的 4 项中，则高亮回退到"More"
- 4. 若记录的值既非已知主页面 tag 也非数字形式的播客 id，则回退加载 Home 页面

## 场景四

### 场景概述

底部导航与侧边栏模式互斥，由"是否启用底部导航"开关决定

### 场景逻辑步骤

- 1. 当用户启用底部导航时，应用构建底部导航栏并锁定/隐藏侧边栏抽屉，主页面切换完全通过底部导航完成
- 2. 当用户关闭底部导航时，底部导航栏整体隐藏，改用侧边栏抽屉进行主页面切换
- 3. 切换该开关后，下次进入主界面时按新模式呈现

## 场景五

### 场景概述

"底部导航基础结构"功能的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下入口: 底部导航栏的 4 个主页面入口与"More"溢出入口；主内容区的主页面切换
- 2. **不作用于** 以下位置: 各主页面内部的内容列表、筛选、排序等交互（由各页面自行处理）；播放页、单集详情页、播客详情页等次级页面不通过底部导航切换
- 3. 底部导航的 4 个主页面组成可由用户通过 More 菜单中的"Customize navigation"调整顺序或隐藏某项；调整后默认的 Home/Queue/Inbox/Subscriptions 组合可能改变

> [偏差] 需求将 4 个主页面固定列为 Home、Queue、Inbox、Subscriptions，但实际 4 个主页面由用户可见导航项顺序决定（默认顺序下为这 4 项，用户可通过"自定义导航"调整组合与顺序）。需求未提及底部导航与侧边栏的互斥关系，实际二者由"启用底部导航"开关二选一。

<!-- REQ-003: More溢出菜单 -->
# More溢出菜单SPEC

用户点击底部导航的 More 入口后，在当前页面之上弹出溢出菜单，菜单列出未直接展示在底部导航的主页面入口及固定的自定义、设置入口

## 场景一

### 场景概述

用户在任意当前主页面点击底部 More，应用弹出覆盖在当前页面之上的浮层菜单

### 场景逻辑步骤

- 1. 用户在 Home、Queue、Inbox 或 Subscriptions 任一主页面（或任意当前页面）点击底部导航最右侧的"More"入口
- 2. 应用不切换主内容区，当前主页面高亮保持不变
- 3. 在底部导航栏上方、靠右侧位置弹出一个浮层菜单，菜单覆盖在当前页面之上，当前页面内容不被替换
- 4. 浮层为模态：用户点击菜单外区域或返回键可关闭浮层，关闭后回到原当前页面

## 场景二

### 场景概述

More 菜单展示条目内容，每行包含图标与文字，集合为"未被底部导航占用的导航项 + 自定义导航 + 设置"

### 场景逻辑步骤

- 1. More 菜单列表每行左侧显示一个图标、右侧显示对应文字
- 2. 在默认底部导航配置（Home、Queue、Inbox、Subscriptions 占用 4 个槽位）下，菜单依次列出：Episodes、Downloads、Playback history、Favorites、Statistics、Add podcast
- 3. 菜单末尾固定追加两项："Customize navigation"（自定义导航）与"Settings"（设置）
- 4. 菜单图标与文字清晰可见，文字使用各页面的完整名称

## 场景三

### 场景概述

用户点击 More 菜单中的导航类条目（如 Episodes、Add podcast），切换到对应主页面

### 场景逻辑步骤

- 1. 用户点击 More 菜单中的某个导航类条目（例如 Add podcast）
- 2. 浮层菜单关闭
- 3. 主内容区切换到该条目对应的主页面，底部导航高亮回退到"More"（因该主页面未在底部 4 槽位中）
- 4. 应用记录该主页面为"当前主页面"

## 场景四

### 场景概述

用户点击 More 菜单中的"Customize navigation"，调整底部导航的组成

### 场景逻辑步骤

- 1. 用户点击 More 菜单中的"Customize navigation"条目
- 2. 浮层菜单关闭，弹出"自定义导航"对话框
- 3. 用户在对话框中调整可见导航项的顺序或隐藏/显示某项后确认
- 4. 底部导航 4 个槽位按新配置重建；下次打开 More 菜单时，其条目集合相应变化（被移入底部 4 槽位的项从 More 菜单消失，被移出的项进入 More 菜单）

## 场景五

### 场景概述

用户点击 More 菜单中的"Settings"，进入设置页面

### 场景逻辑步骤

- 1. 用户点击 More 菜单末尾的"Settings"条目
- 2. 浮层菜单关闭
- 3. 应用打开设置页面（设置页面的内容详见对应设置页规格）

## 场景六

### 场景概述

"More 溢出菜单"功能的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下入口: 底部导航"More"入口触发的浮层菜单；菜单内条目的点击跳转
- 2. **不作用于** 以下位置: 已放入底部导航 4 槽位的主页面入口（这些入口直接由底部导航切换，不进入 More 菜单）；侧栏模式下没有 More 入口（侧栏直接列出全部导航项）
- 3. More 菜单的导航类条目集合随底部导航配置动态变化，仅"Customize navigation"与"Settings"两项固定存在

> [偏差] 需求用"至少包含"列举菜单项，但代码中导航类条目（Episodes、Downloads 等）是"未被底部 4 槽位占用的导航项"，即用户通过自定义导航调整后，这些条目会相应进入或离开 More 菜单；仅"自定义导航"与"设置"两项为固定项。默认配置下菜单恰好包含需求列举的全部项。需求未提及菜单的宽度与弹出位置，实际固定宽度并靠右下角弹出。

<!-- REQ-004: More跳转设置页 -->
# More跳转设置页SPEC

用户通过底部导航的 More 菜单进入设置页面，设置页面提供返回按钮、标题、搜索框与设置分类列表，点击分类进入二级设置页，返回可回到原页面

## 场景一

### 场景概述

用户点击底部 More，在弹出菜单中选择 Settings，进入设置页面

### 场景逻辑步骤

- 1. 用户在任意当前主页面点击底部导航的"More"入口
- 2. 在弹出的 More 菜单中点击末尾的"Settings"项
- 3. More 菜单关闭，应用打开独立的设置页面
- 4. 设置页面顶部显示返回按钮、标题"Settings"、搜索框，下方显示设置分类列表

## 场景二

### 场景概述

设置页面展示主要设置分类列表

### 场景逻辑步骤

- 1. 设置页面的分类列表包含以下主要分类：User interface（用户界面）、Playback（播放）、Downloads（下载）、Synchronization（同步）、Backup & restore（备份与恢复）、Notifications（通知）
- 2. 在特定条件下，列表还会显示 Parental Controls（家长控制）：仅当设备为受监管的子设备、调试构建或已设置家长控制密码时可见
- 3. 列表末尾追加 Project 分类，包含 Documentation、Visit user forum、Contribute、Report bug、About 等项目入口
- 4. 页面可滚动浏览全部分类

## 场景三

### 场景概述

用户点击设置分类，进入对应的二级设置页

### 场景逻辑步骤

- 1. 用户点击分类列表中的某一分类（例如 User interface）
- 2. 主内容区切换为该分类的二级设置页，顶部标题更新为该分类名称（如"User interface"）
- 3. 该次切换被记入返回栈，以便逐级返回
- 4. 特例：点击 Notifications 时，若系统支持，应用直接打开系统级通知设置而非应用内二级页
- 5. 特例：点击 Parental Controls 时，若已设置家长控制密码，先弹出密码验证，验证通过后才进入二级页

## 场景四

### 场景概述

用户在设置页使用搜索框检索设置项

### 场景逻辑步骤

- 1. 用户点击设置页面顶部的搜索框，输入关键词
- 2. 搜索框在已索引的全部二级设置页中检索匹配的设置项，并展示结果
- 3. 结果带有面包屑路径，标明该设置项所属的分类层级
- 4. 用户点击某结果，应用直接跳转到该设置项所在的二级页并高亮该设置项

## 场景五

### 场景概述

用户在设置页点击返回，逐级回到上级直至回到进入设置前的原页面

### 场景逻辑步骤

- 1. 用户点击设置页面顶部的返回按钮（或系统返回键）
- 2. 若当前在某个二级分类页，则返回到设置分类列表，标题恢复为"Settings"
- 3. 若当前已在设置分类列表顶层，则关闭设置页面，回到进入设置前的原当前主页面
- 4. 返回过程中收起软键盘

## 场景六

### 场景概述

"More 菜单跳转设置页"功能的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下入口: 底部 More 菜单的 Settings 项；侧边栏模式的 Settings 项；睡眠定时器等带"直达某二级页"参数的内部入口
- 2. **不作用于** 以下位置: 主界面的主页面切换（设置页是独立页面，不通过底部导航切换）；各二级设置页内部具体开关的行为（由各自规格承接）
- 3. 设置页可被侧边栏、睡眠定时器等多个入口打开，且支持携带参数直达某个二级设置页

# 订阅后数据刷新SPEC

用户完成一次订阅后，Subscriptions 列表出现该播客，Home 从空库状态切换为有内容状态并刷新各分区，底部导航 Inbox 角标同步更新

- 3. 应用向内部广播"订阅源列表变更"事件，携带该播客标识

Subscriptions 页面刷新后，新订阅的播客出现在订阅列表中

Home 页面从空库欢迎状态切换为分区内容展示

### 场景概述

Home 各分区在订阅后分别刷新，展示订阅相关的单集摘要

## 场景五

### 场景逻辑步骤

# 单集列表项SPEC

在"播客详情页"和"Episodes 总列表页"(以及复用同一列表项结构的其它页面)中, 以统一格式展示单个单集的信息与状态, 支持点击进入单集详情、长按进入多选, 列表可滚动并按需加载更多。

### 场景概述

列表项展示单个单集的完整信息与状态

### 场景逻辑步骤

- 1. 用户进入"播客详情页"或"Episodes 总列表页"(或其它展示单集列表的页面)
- 2. 每个单集列表项自左向右依次渲染: 拖拽手柄(默认隐藏, 仅在队列等需要拖拽的场景显示)、封面(带圆角的方形封面图, 封面加载前显示播客标题作为占位文字)、主信息列、次级动作按钮
- 3. 主信息列顶部状态行可包含以下图标(按状态显隐): "新单集"图标(该单集未被处理过时显示)、"视频"图标(单集媒体为视频时显示)、"收藏"图标(单集已被收藏时显示)、"在队列"图标(单集已加入播放队列时显示), 随后显示发布日期与单集大小
- 4. 主信息列中部显示单集标题(可换行, 标题过长时截断)
- 5. 主信息列底部进度行显示: 当前播放位置文本、进度条、时长文本(未播放或未下载时, 进度行隐藏)
- 6. 单集大小在已知时显示(如"10 MB"), 未知时该位置留空
- 7. 时长在已知时显示, 未知时隐藏

## 场景二

### 场景概述

正在播放或已播放过一部分的单集, 列表项显示进度与剩余时长; 已播放完成的单集整行半透明

### 场景逻辑步骤

- 1. 列表项对应的单集处于"正在播放"或"已播放部分"状态
- 2. 进度行显示当前播放位置文本、进度条(按已播放比例填充)、时长文本
- 3. 若用户开启了"显示剩余时间"偏好, 时长文本切换为剩余时长的负值形式(如"-00:12:34"), 否则显示单集总时长
- 4. 当前正在播放的单集列表项整行处于高亮激活状态
- 5. 若该单集已播放完成(已播放标记为真), 整行透明度降低为半透明, 以视觉区分已听完的单集

## 场景三

### 场景概述

下载中或已下载的单集, 次级动作按钮反映下载状态

### 场景逻辑步骤

- 1. 列表项对应的单集媒体处于"下载中"状态
- 2. 次级动作按钮显示为带百分比的进度环, 百分比随下载进度实时更新

## 场景四

### 场景逻辑步骤

- 1. 列表展示期间, 单集状态发生变更(如: 收藏/取消收藏、加入/移出队列、标记为已播放、下载进度变化、下载完成、开始或停止播放、被标记为新单集或从新单集中移除)
- 2. 仅受影响的单集列表项被局部刷新(不重绘整列表)
- 3. 刷新后, 状态图标(收藏/队列/新单集/视频)、进度行、次级动作按钮、整行透明度与高亮状态, 均与新状态一致

## 场景五

用户长按单集列表项弹出上下文菜单并可进入多选模式

### 场景逻辑步骤

- 4. 多选模式下, 用户继续点击其它列表项可逐个勾选或取消勾选(不再触发进入详情)

# 下载单集SPEC

用户可在单集详情页、单集列表(多选批量、侧滑)或由自动下载后台触发, 对单个或多个单集发起下载; 下载中显示进度并可取消, 下载完成后单集标记为已下载、可用于离线播放等场景。移动网络受限时先经用户确认。

## 场景一

### 场景概述

用户在单集详情页点击"下载", 在允许网络下立即开始下载

### 场景逻辑步骤

- 1. 用户进入单集详情页, 该单集未下载且不在下载中
- 2. 详情页右下次操作按钮显示为"下载"
- 3. 用户点击"下载"按钮
- 4. 若当前网络允许下载(如在 Wi-Fi 下, 或已开启移动数据下载), 立即开始下载, 按钮切换为"取消下载"并叠加下载进度环
- 5. 若该单集无媒体或已下载, 点击不产生任何效果(不重复下载)

## 场景二

### 场景概述

移动网络受限时, 用户点击"下载"弹出确认对话框

### 场景逻辑步骤

- 1. 用户在详情页点击"下载", 但当前网络为受限的移动网络(且未开启移动数据下载)
- 2. 弹出确认对话框, 标题为"移动网络下载确认", 提示当前为移动网络, 并提供三个选项: "稍后下载"(等设备条件满足再下载)、"本次允许"(本次立即下载)、"取消"
- 3. 用户选择"稍后下载": 按设备条件排队下载, 按钮切换为"取消下载"
- 4. 用户选择"本次允许": 立即开始下载, 且在随后一段约定时间内再次下载其它单集时不再弹出此对话框
- 5. 用户选择"取消"或点击对话框外部: 不开始下载, 按钮保持"下载"

## 场景三

### 场景概述

下载中显示进度环, 排队等待时显示不确定动画

### 场景逻辑步骤

- 1. 单集进入下载流程后, 详情页次操作按钮与所在列表项的次级动作区显示下载进度环
- 2. 若该单集正在实际下载(运行中), 进度环按下载百分比填充
- 3. 若该单集在下载队列中排队等待, 进度环显示为不确定状态的动画
- 4. 下载过程中, 进度随事件实时刷新(无需用户手动刷新)

## 场景四

### 场景概述

用户在下载过程中取消下载

### 场景逻辑步骤

- 1. 单集处于下载中(运行中或排队等待)
- 2. 详情页次操作按钮显示为"取消下载"
- 3. 用户点击"取消下载", 终止该单集的下载任务
- 4. 同时关闭该单集的自动下载标记(后续不会再自动下载它)
- 5. 按钮恢复为"下载", 进度环消失

## 场景五

### 场景概述

下载完成后, 单集标记为已下载并可用于离线场景

### 场景逻辑步骤

- 1. 下载任务成功完成
- 2. 该单集的媒体被标记为"已下载", 并记录下载完成时间, 持久化到数据库
- 3. 详情页与列表项的次操作按钮切换为"删除下载"(已下载状态)
- 4. 该单集此后可用于离线播放、出现在下载页等场景

## 场景六

### 场景概述

用户在单集列表中通过多选批量下载

### 场景逻辑步骤

- 1. 用户在单集列表(详情页列表/Episodes/搜索结果/下载页等)长按进入多选模式
- 2. 选中一个或多个未下载的单集
- 3. 在多选动作栏中选择"下载"
- 4. 所选单集按设备条件依次触发下载, 各自进入下载中状态(显示进度环)
- 5. 批量下载不弹出单集级的移动网络确认(由批量上下文与设备下载约束统一决定)

## 场景七

### 场景概述

用户在列表项上侧滑触发下载

### 场景逻辑步骤

- 1. 用户在支持侧滑的列表项上向"下载"方向滑动
- 2. 若该单集无媒体, 提示"无媒体", 不下载
- 3. 若该单集已下载, 提示"已下载", 不重复下载
- 4. 否则触发与详情页相同的下载流程(含移动网络确认逻辑), 进入下载中状态

## 场景八

### 场景概述

自动下载在后台触发单集下载

### 场景逻辑步骤

- 1. 当订阅源刷新发现新单集, 且对应播客设置开启了自动下载、且满足自动下载条件时
- 2. 系统在后台对满足条件的单集发起下载(无需用户主动点击)
- 3. 下载中状态同样在列表项与详情页上显示进度环
- 4. 下载完成后与手动下载一致, 标记为已下载

## 场景九

### 场景概述

"下载单集"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下入口的单集下载: 单集详情页下载按钮; 单集列表的多选批量下载; 列表项侧滑下载; 下载日志中的失败重试; 后台自动下载
- 2. **不作用于** 以下位置/关系: 收藏页(下载与收藏相互独立, 下载不改变收藏状态); 桌面小部件(不触发下载); 播放队列(下载完成本身不会自动把单集加入队列, 加入队列是独立动作)
- 3. "下载完成后出现在下载页""下载日志与失败详情"由各自独立规格约束; 本规格仅描述下载这一动作与下载中/完成/取消的状态

# Queue数据来源SPEC

播放队列(Queue)是一个持久化的有序单集列表, 其数据来源于: 用户主动"加入队列"操作(详情页菜单、列表上下文菜单、列表多选批量、列表侧滑), 以及后台联动入队(自动下载、播放服务、订阅源新单集按设置)。加入队列时按"入队位置"偏好计算插入位置, 并为单集打上队列标签。手动下载本身不会自动把单集加入队列。

## 场景一

### 场景概述

用户在单集详情页通过"更多"菜单将单集加入队列

### 场景逻辑步骤

- 1. 用户进入单集详情页, 该单集有媒体且当前不在队列中
- 2. 用户打开"更多"溢出菜单, 选择"加入队列"(不可加入时不显示该项)
- 3. 该单集按"入队位置"偏好(如末尾或当前播放项之后)插入队列, 写入数据库并打上队列标签
- 4. 若该单集原为"新单集"状态, 同时被标记为未播放并移出新单集状态
- 5. 队列页面与各列表项的"在队列"图标随后刷新; 该单集已可被队列场景使用
- 6. 对已在队列中的单集再次"加入队列", 不会改变其位置(保持原位)

## 场景二

### 场景概述

用户在单集列表项上下文菜单中将单集加入队列

### 场景逻辑步骤

- 1. 用户在单集列表(详情页列表/Episodes/搜索结果/下载页等)长按某个单集
- 2. 在弹出的上下文菜单中选择"加入队列"
- 3. 该单集按入队位置偏好加入队列(同场景一的写入与状态联动)

## 场景三

### 场景概述

用户在列表多选模式下批量加入队列

### 场景逻辑步骤

- 1. 用户在单集列表长按进入多选模式
- 2. 选中一个或多个单集(可能有媒体也可能无媒体)
- 3. 在多选动作栏中选择"加入队列"
- 4. 所选单集中"有媒体且不在队列中"的依次加入队列(按入队位置偏好连续插入)
- 5. 无媒体的项、已在队列中的项被跳过, 不影响其它项入队

## 场景四

### 场景概述

用户在列表项上侧滑将单集加入队列

### 场景逻辑步骤

- 1. 用户在支持侧滑的列表项上向"加入队列"方向滑动
- 2. 该单集按入队位置偏好加入队列(同场景一的写入与状态联动)

## 场景五

### 场景概述

自动下载在后台将单集加入队列(下载与入队联动)

### 场景逻辑步骤

- 1. 当自动下载对满足条件的单集发起下载时, 该单集同时被加入队列
- 2. 即"被自动下载的单集会出现在队列中"
- 3. 队列页面随后刷新显示该单集

## 场景六

### 场景概述

播放服务在播放流程中将单集加入队列

### 场景逻辑步骤

- 1. 播放服务在需要时(如为保证连续播放、把可播放项纳入队列)将单集加入队列
- 2. 该单集按入队位置偏好加入队列, 队列页面随后刷新

## 场景七

### 场景概述

订阅源刷新发现新单集时按用户设置入队

### 场景逻辑步骤

- 1. 订阅源刷新解析出新的单集
- 2. 按用户对"新单集处理方式"的设置, 满足条件的新单集被加入队列
- 3. 队列页面随后刷新显示该单集

## 场景八

### 场景概述

用户进入 Queue 页面查看队列数据与统计

### 场景逻辑步骤

- 1. 用户对若干单集执行上述任一加入队列操作后, 点击底部"Queue"
- 2. Queue 页面从数据库读取有序队列并展示, 被加入的单集出现在列表中
- 3. 顶部统计(队列单集数量、剩余播放总时长)随队列内容更新
- 4. 队列为空时显示空态(详见队列展示规格)

## 场景九

### 场景概述

队列内容的实时刷新(事件驱动)

### 场景逻辑步骤

- 1. 队列展示期间, 任何加入/移出/移动/清空队列的操作都会广播队列事件
- 2. Queue 页面与各处的"在队列"图标接收事件后局部刷新, 无需用户手动刷新或重新进入页面
- 3. 单集的播放进度变化也会刷新队列中该项的剩余时长

## 场景十

### 场景概述

"Queue 数据来源"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **队列数据仅来源于** 以下操作: 用户主动"加入队列"(详情页菜单/列表上下文菜单/多选批量/侧滑); 自动下载入队(后台); 播放服务入队(后台); 订阅源新单集按设置入队(后台)
- 2. **不来源于** 以下操作: 手动下载(手动下载不会自动把单集加入队列, 除非走自动下载流程); 收藏(收藏与队列相互独立, 收藏不写入队列); 标记为已播放/未播放(不改变队列)
- 3. 队列的"展示""移除""排序与移动""清空"由各自独立规格约束; 本规格仅描述数据如何进入队列
- 4. [偏差] 若用户通过"手动下载"期望单集出现在队列, 实际不会出现; 仅"加入队列"或自动下载/播放/按设置入队才会使单集出现在队列

# Queue页面展示SPEC

Queue 页面展示待播放单集的有序列表, 顶部统计条显示单集总数量与剩余播放总时长; 每个队列项展示封面、发布日期、大小、标题、时长/进度, 并提供拖拽手柄(用于手动排序)与侧滑移除; 队列为空时显示空态; 内容随事件实时刷新。

## 场景一

### 场景概述

用户进入 Queue 页面查看队列列表与顶部统计

### 场景逻辑步骤

- 1. 用户点击底部"Queue"(或从首页队列分区、抽屉等入口进入)
- 2. Queue 页面从数据库读取有序队列并展示, 加载期间显示加载指示
- 3. 顶部统计条显示: 队列单集总数(如"N 个单集")与剩余播放总时长
- 4. 剩余时长按各单集"时长减去已播放位置"累加; 若开启了"按播放速度折算时间", 则按各单集的播放速度折算后累加

队列项展示封面、日期、大小、标题与时长/进度

### 场景逻辑步骤

- 1. 每个队列项自左向右展示: 拖拽手柄、封面(圆角方形)、主信息列、次级动作区

## 场景三

拖拽手柄用于手动排序, 且其可见性受队列状态控制

### 场景概述

队列为空时显示空态

### 场景逻辑步骤

- 1. 当队列中没有任何单集时, 列表区显示空态(图标、标题与说明), 提示队列无内容

队列内容与统计随事件实时刷新

用户在 Queue 页面下拉刷新订阅源

- 2. 刷新运行期间显示刷新指示

# Queue移除单集SPEC

用户可从 Queue 中移除某个或多个单集; 移除后该单集从队列消失、顶部统计同步更新, 其余队列项顺序保持不变; 移除不删除单集本身, 也不影响其下载文件与收藏状态。移除入口以侧滑为主, 辅以上下文菜单、多选批量与后台联动; 侧滑移除支持撤销。

### 场景概述

用户在 Queue 页面侧滑移除单个队列项

- 1. 用户在 Queue 页面将某个队列项向左或向右滑动
- 2. 该单集从队列移除: 队列列表中该项消失, 队列顺序持久化
- 3. 顶部统计(单集数量、剩余播放时长)立即按新队列重新计算并更新
- 4. 其余队列项的相对顺序保持不变
- 5. 若因移除导致该项在当前筛选下不再可见, 显示"已从队列移除"提示并提供"撤销"

## 场景二

### 场景概述

用户长按队列项通过上下文菜单移除

### 场景逻辑步骤

- 1. 用户在 Queue 页面长按某个队列项
- 2. 弹出上下文菜单, 选择"从队列移除"
- 3. 该单集从队列移除, 顶部统计与列表同步更新

## 场景三

### 场景概述

用户在 Queue 页面多选批量移除

- 2. 通过浮动动作菜单选择"从队列移除"

## 场景四

### 场景概述

用户在单集详情页或其它单集列表中"从队列移除"

### 场景逻辑步骤

- 1. 用户进入某个已在队列中的单集的详情页(或任意单集列表的上下文菜单)
- 2. 在"更多"菜单或上下文菜单中选择"从队列移除"(仅当该单集在队列中时该选项才显示)

### 场景概述

播放完成后自动移出队列(后台联动)

### 场景逻辑步骤

- 1. 当队列中某个单集播放完成时, 按用户的"播放完成行为"设置, 该单集自动移出队列

- 3. 该单集本身不被删除, 仍保留在播客详情与 Episodes 中

## 场景六

同步与下载取消导致的后台移除

- 2. 当取消某个单集的下载时, 按流程联动将该单集移出队列
- 3. 移除后队列列表与统计同步更新

## 场景七

### 场景概述

移除队列后的自动下载重评估

### 场景概述

"Queue 移除单集"的作用范围与不作用范围

- 2. **不作用于** 以下对象: 单集本身(仍保留在播客详情与 Episodes 中); 已下载的媒体文件(不删除下载); 收藏状态(不影响是否收藏); 播放历史(不影响历史记录)

- 4. [偏差] 需求步骤描述"点击移除按钮", 实际主入口为侧滑(含撤销), 另有上下文菜单与多选批量; 队列项上无常驻移除按钮

# Queue排序与移动SPEC

用户可在 Queue 页面调整队列顺序: 拖拽重排、将单集移到顶部或底部、按排序规则整体排序(可保持排序)、锁定队列。所有调整都写入数据库, 刷新或切换页面后顺序保持。拖拽在队列锁定或开启保持排序时禁用。

## 场景一

### 场景概述

用户通过拖拽手柄手动调整队列顺序

### 场景逻辑步骤

- 1. 队列处于"未锁定且未开启保持排序"状态, 各队列项左侧显示拖拽手柄
- 2. 用户按住某个队列项的拖拽手柄(或封面左半区)上下拖动
- 3. 拖动过程中, 该项实时跟随到新位置, 其余项相应让位
- 4. 用户松手后, 新顺序写入数据库并持久化
- 5. 顶部统计不变(顺序变化不影响数量与总时长), 列表保持新顺序
- 6. 刷新或切换页面后再回来, 顺序保持

## 场景二

### 场景概述

用户将单个队列项移到顶部或底部

### 场景逻辑步骤

- 1. 用户长按某个队列项, 弹出上下文菜单
- 2. 若该单集不在队列首位, 菜单显示"移到顶部"; 若不在末位, 显示"移到底部"(已是首/末位时对应项隐藏)
- 3. 用户选择"移到顶部"或"移到底部", 该项立即移到队列首位或末位
- 4. 新顺序写入数据库并持久化, 列表与统计同步刷新
- 5. 当队列开启了"保持排序"时, 这两项菜单都隐藏(顺序由排序规则决定)

## 场景三

### 场景概述

用户多选批量移到顶部或底部

### 场景逻辑步骤

- 1. 用户在 Queue 页面长按进入多选模式, 选中一个或多个队列项
- 2. 系统根据所选位置判定"移到顶部""移到底部"是否可用(不可用则隐藏对应动作)
- 3. 用户选择"移到顶部"或"移到底部", 所选项按原相对顺序整体移到队列首或尾
- 4. 新顺序写入数据库并持久化, 列表与统计同步刷新

## 场景四

### 场景概述

用户打开排序对话框按规则排序队列

### 场景逻辑步骤

- 1. 用户点击 Queue 页面工具栏菜单中的"排序"
- 2. 弹出排序对话框, 列出可用排序规则, 可包括: 日期(由新到旧/由旧到新)、标题(A-Z/Z-A)、时长(短到长/长到短)、播客名(A-Z/Z-A)、随机、智能打乱(由新到旧/由旧到新)
- 3. 当前生效的排序规则处于选中状态
- 4. 用户选择一项, 队列立即按该规则重排, 新顺序写入数据库
- 5. 选择"随机"时, "保持排序"选项自动关闭并禁用(随机无需保持)

## 场景五

### 场景概述

用户开启"保持排序", 使后续加入的单集自动按规则排序

### 场景逻辑步骤

- 1. 用户在排序对话框中勾选"保持排序"
- 2. 当前队列按所选规则排序, 且该规则与"保持排序"状态被保存
- 3. 此后新加入队列的单集自动按该规则插入到对应位置, 而非简单追加到末尾
- 4. 开启保持排序后, 拖拽手柄隐藏, 不可手动拖拽重排; "移到顶部/底部"菜单也隐藏
- 5. 用户再次打开排序对话框可切换规则或取消"保持排序", 取消后手柄恢复

## 场景六

### 场景概述

用户锁定队列以防止误操作

### 场景逻辑步骤

- 1. 用户点击 Queue 页面工具栏菜单中的"锁定"(首次可能弹出锁定说明提示)
- 2. 队列进入锁定状态, 拖拽手柄隐藏, 不可手动拖拽重排
- 3. 锁定不影响排序对话框与"保持排序"的使用
- 4. 用户再次点击"锁定"解锁, 拖拽手柄恢复(若同时未开启保持排序)

## 场景七

### 场景概述

顺序变更后跨页面与刷新的持久性

### 场景逻辑步骤

- 1. 用户执行任一重排(拖拽/移到顶底/排序)后, 新顺序立即写入数据库
- 2. 用户切换到其它页面再回到 Queue, 顺序保持不变
- 3. 用户下拉刷新或应用重启后, 顺序仍保持为上次写入的顺序
- 4. 仅当用户再次执行重排或清空队列时, 顺序才改变

## 场景八

### 场景概述

"Queue 排序与移动"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** Queue 页面的队列顺序: 拖拽重排、移到顶部/底部、排序(含保持排序)、锁定对拖拽的影响
- 2. **不作用于** 以下列表的顺序: 播客详情页单集列表、Episodes 总列表(它们有各自的排序规则, 与队列顺序相互独立); 收藏列表、下载列表、收件箱(不读取队列顺序)
- 3. 队列的"展示""数据来源""移除""清空"由各自独立规格约束; 本规格仅描述顺序调整

# 清空QueueSPEC

用户可在 Queue 页面通过工具栏菜单清空整个播放队列, 清空前需二次确认; 清空后队列显示空态、顶部统计归零, 相关"在队列"标记同步移除; 清空只作用于队列关系, 不删除订阅、单集元数据、下载文件、收藏或播放历史; 清空不可撤销。

## 场景一

### 场景概述

用户从工具栏菜单清空队列并确认

### 场景逻辑步骤

- 1. 用户进入 Queue 页面(队列中已有至少一个单集)
- 2. 用户点击顶部工具栏菜单, 选择"清空队列"
- 3. 弹出确认对话框, 提示"清空队列"及确认信息
- 4. 用户点击"确认"
- 5. 队列存储被清空(所有单集与队列的关联被移除), 写入数据库
- 6. Queue 页面立即刷新为空态: 列表区显示空态图标与提示, 顶部统计的单集数量与剩余时长归零

### 场景概述

"清空 Queue"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 队列关系: 清空队列存储、移除所有单集与队列的关联、刷新队列页面为空态、同步"在队列"标记
- 2. **不作用于** 以下对象: 单集本身(仍保留在播客详情与 Episodes); 订阅关系; 已下载的媒体文件; 收藏状态; 播放历史; 当前播放
- 3. 队列的"展示""数据来源""移除""排序与移动"由各自独立规格约束; 本规格仅描述"清空"这一动作

# Inbox数据含义SPEC

Inbox 展示订阅源刷新后发现的、尚未被用户处理的新单集(即"新单集"标记为真的单集)。它不是所有单集列表, 也不是下载列表、收藏列表或播放队列。有新单集时显示新单集列表, 没有时显示合理空态; 单集的"新"标记会因用户处理(标记已播放、移出收件箱、加入队列等)而被移除, 随之从 Inbox 消失, 但单集本身仍保留在播客详情与 Episodes 中。

## 场景一

### 场景概述

Inbox 的数据范围: 仅展示新单集

### 场景逻辑步骤

- 1. 用户订阅播客并刷新订阅源后, 系统为发现的新单集设置"新单集"标记
- 2. 用户点击底部"Inbox"
- 3. Inbox 页面从数据库加载所有"新单集"标记为真的单集, 按排序规则分页展示
- 4. 列表中每项展示封面、标题、日期、大小、时长/进度等(复用单集列表项), 并显示"新单集"状态图标
- 5. 单集数量超过一屏时, 滚动接近底部自动加载下一批

## 场景二

### 场景概述

Inbox 不是 Queue、Downloads、Favorites 或全部 Episodes

### 场景逻辑步骤

- 1. Inbox 的数据源严格限定为"新单集"筛选, 与以下列表在数据语义上不同:
  - 1.1 不同于"播放队列": 一个单集被加入队列后, 其"新单集"标记会被移除, 随之从 Inbox 消失(但仍出现在 Queue)
  - 1.2 不同于"下载列表": 已下载但"新单集"标记已移除的单集不出现在 Inbox(出现在 Downloads)
  - 1.3 不同于"收藏列表": 已收藏但"新单集"标记已移除的单集不出现在 Inbox(出现在 Favorites)
  - 1.4 不同于"全部 Episodes": Episodes 展示所有订阅单集(不论是否为新), Inbox 仅展示新单集
- 2. 当一个单集同时是"新"且"已下载/在队列/已收藏"时(标记相互独立), 它会同时出现在 Inbox 与对应页面, 这属于多重标记共存, 不是误显示

## 场景三

### 场景概述

没有新单集时显示合理空态

### 场景逻辑步骤

- 1. 当没有任何"新单集"标记为真的单集时(如用户已处理完所有新单集, 或订阅源未发现新单集)
- 2. Inbox 页面显示空态(图标、标题与说明), 提示暂无新单集
- 3. 订阅源再次刷新发现新单集后, 空态自动切换为新单集列表

## 场景四

### 场景概述

新单集因用户处理而从 Inbox 消失

### 场景逻辑步骤

- 1. 用户对某个新单集执行以下任一处理:
  - 1.1 标记为已播放
  - 1.2 从收件箱移除(侧滑或菜单)
  - 1.3 加入播放队列
- 2. 该单集的"新单集"标记被移除
- 3. 该单集随之从 Inbox 列表消失(因不再满足"新单集"筛选)
- 4. 该单集本身不被删除, 仍保留在对应播客的详情页单集列表与 Episodes 总列表中
- 5. 顶部数量与列表立即刷新

## 场景五

### 场景概述

用户点击 Inbox 中的单集进入详情

### 场景逻辑步骤

- 1. 用户点击 Inbox 列表中的某个新单集(非多选模式)
- 2. 进入单集详情页, 以当前 Inbox 列表为分页上下文
- 3. 该操作不修改单集的"新单集"标记(单集仍在 Inbox 中, 除非用户在详情页执行处理)

## 场景六

### 场景概述

用户对 Inbox 列表排序

### 场景逻辑步骤

- 1. 用户点击 Inbox 工具栏菜单中的"排序"
- 2. 弹出排序对话框, 列出可用排序规则(日期由旧到新、时长由短到长)
- 3. 当前生效的排序规则处于选中状态
- 4. 用户选择一项, 列表按新规则重新排序, 偏好被保存

## 场景七

### 场景概述

用户一次性将所有新单集移出 Inbox

### 场景逻辑步骤

- 1. 用户点击 Inbox 工具栏菜单中的"全部移出收件箱"
- 2. 若此前已选择"不再询问", 直接执行; 否则弹出确认对话框, 提示将移出所有新单集, 并提供"不再询问"复选框
- 3. 用户确认后, 清除所有单集的"新单集"标记, 提示已完成
- 4. Inbox 列表清空, 显示空态; 原单集仍保留在播客详情与 Episodes 中
- 5. 用户取消则不做任何变更

## 场景八

### 场景概述

用户在 Inbox 列表项上侧滑快速处理

### 场景逻辑步骤

- 1. 用户在某个新单集上向左或向右滑动, 触发该方向配置的快捷动作(如标记为已读/移出收件箱)
- 2. 该单集的"新单集"标记被移除, 随之从 Inbox 消失
- 3. 原单集仍保留在播客详情与 Episodes 中
- 4. (侧滑移出的完整行为, 包括撤销, 由"Inbox 单集处理"规格约束)

## 场景九

### 场景概述

Inbox 随事件实时刷新

### 场景逻辑步骤

- 1. Inbox 展示期间, 订阅源刷新发现新单集时, 新单集加入 Inbox 列表
- 2. 单集被处理(标记已播放/移出/加入队列)后, 对应项从 Inbox 消失
- 3. 无需用户手动刷新或重新进入页面

## 场景十

### 场景概述

"Inbox 数据含义"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **Inbox 仅包含** "新单集"标记为真的订阅单集(订阅源刷新发现、用户尚未处理)
- 2. **Inbox 不包含** 以下数据: 播放队列(队列数据); 下载列表(已下载数据); 收藏列表(收藏数据); 全部 Episodes(非新的历史单集)
- 3. **不作用于** 桌面小部件(不渲染 Inbox 列表)
- 4. Inbox 的"单集处理(移出/标记/撤销)"完整流程由"Inbox 单集处理"规格约束; 列表项展示归单集列表项规格、详情页归单集详情页规格; 本规格仅定义 Inbox 的数据语义与边界

# Inbox 单集处理SPEC

用户在 Inbox 中查看新单集，并可把单集从 Inbox 移除；移除后该单集不再显示在 Inbox，但仍保留在播客详情与 Episodes 中。支持单集移除与批量清空两种粒度。

## 场景一

### 场景概述

用户在 Inbox 列表中把某条单集向左滑出，将其从 Inbox 移除，并可在约 2 秒内撤销

### 场景逻辑步骤

- 1. 用户进入底部"Inbox"
- 2. 用户对列表中某条单集执行向左滑出操作
- 3. 该单集立即从 Inbox 列表消失
- 4. 弹出浮动提示"Removed from inbox"，并显示"撤销"按钮，约 2 秒后自动消失
  - 4.1 若用户在该浮动提示消失前点击"撤销"，则该单集恢复为移除前的"新单集"状态，重新出现在 Inbox 原位置
  - 4.2 若用户未点击"撤销"，约 2 秒后该移除最终落定，单集保持已移出 Inbox 的状态

## 场景二

### 场景概述

用户通过单集选项菜单把单集从 Inbox 移除（无撤销）

### 场景逻辑步骤

- 1. 用户在 Inbox 或其它含单集选项菜单的列表中，打开某条单集的选项菜单
- 2. 用户点击菜单中的"Remove from inbox"项
- 3. 该单集的"新单集"标记被清除，从 Inbox 列表消失
- 4. 该路径不弹出"撤销"浮动提示

## 场景三

### 场景概述

用户在 Inbox 多选模式下批量移除多条单集

### 场景逻辑步骤

- 1. 用户进入底部"Inbox"，进入多选模式并选中两条及以上单集
- 2. 用户点击批量动作中的"Remove from inbox"
- 3. 选中单集中所有仍为"新单集"的项被批量清除"新单集"标记
- 4. 弹出数量浮动提示，显示本次被移除的单集数量；该路径不提供单项"撤销"
- 5. 被移除的单集从 Inbox 列表消失

## 场景四

### 场景概述

用户通过 Inbox 工具栏菜单一次性清空 Inbox 全部单集

### 场景逻辑步骤

- 1. 用户进入底部"Inbox"
- 2. 用户打开页面顶部工具栏的溢出菜单，点击"Remove all from inbox"
- 3. 弹出确认对话框，提示"请确认要清空 Inbox 全部单集"，并附带"不再询问"勾选框
- 4. 用户在确认对话框中：
  - 4.1 点击"取消"，关闭对话框，Inbox 内容保持不变
  - 4.2 勾选"不再询问"并点击"确认"，本次清空全部"新单集"标记，并将"不再询问"偏好保存；下次再点击"Remove all from inbox"时不再弹出确认对话框，直接执行清空
  - 4.3 不勾选"不再询问"直接点击"确认"，本次清空全部"新单集"标记，下次仍会弹出确认对话框
- 5. 清空完成后，Inbox 列表变为空态，弹出"Removed all from inbox"浮动提示
- 6. 清空操作仅清除"新单集"标记，不删除任何单集本体

## 场景五

### 场景概述

单集被移出 Inbox 后，原始单集仍保留在播客详情与 Episodes 中，状态在切换页面后保持

### 场景逻辑步骤

- 1. 用户对某条单集执行上述任一移除操作（左滑 / 选项菜单 / 批量 / 清空）
- 2. 该单集的"新单集"标记被清除，但单集本体（封面、标题、发布信息、媒体等）不被删除
- 3. 用户切换到播客详情页或 Episodes 页，该单集仍正常出现在对应列表中
- 4. 用户返回 Inbox，被移除的单集不再出现；切换页面后再回来，状态保持一致

## 场景六

### 场景概述

"Inbox 单集移除"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下界面与数据：底部"Inbox"列表的单集显示（依据"新单集"标记过滤）；首页 Inbox 卡片（同样依据"新单集"标记）；单集列表项的"新单集"角标显示
- 2. **不作用于** 以下界面：播放队列、收藏、下载、播放历史 — 这些列表各自依据独立标记渲染，单集从 Inbox 移除不影响其在这些列表中的存在与状态
- 3. **隐式触发**：当单集因播放完成或手动"标记为已播放"而失去"新单集"标记时，也会自动从 Inbox 消失；当订阅源刷新拉取到新单集时，新单集默认带"新单集"标记并自动进入 Inbox

> [偏差] 需求原文仅描述"对某个单集执行 Remove from inbox"（单集粒度），未提及工具栏"清空全部 Inbox"批量入口与"不再询问"勾选；为完整覆盖代码已发现的全部入口，本 spec 将批量清空纳入场景四。需求未提及撤销机制；代码仅在"左滑"路径提供约 2 秒撤销窗口，"选项菜单"与"多选批量"路径不提供撤销（见场景一与场景二）。需求表述为"移除"，实际不清除单集本体，仅清除其"新单集"标记。

# Favorites 收藏入口SPEC

用户可在单集详情/播放页或其它单集列表入口点击星标（或等价菜单项）收藏单集，收藏后该单集出现在 Favorites 页面；再次操作取消收藏，从 Favorites 移除。支持单集与多选批量两种粒度。

## 场景一

### 场景概述

用户从任一单集粒度入口切换某条单集的收藏状态（加入或移出）

### 场景逻辑步骤

- 1. 用户从以下任一入口对某条单集发起收藏切换：
  - 1.1 在音频播放页顶部工具栏点击星标按钮
  - 1.2 在视频播放页顶部工具栏点击星标按钮
  - 1.3 在单集的选项（三点）菜单中点击"Add to favorites"或"Remove from favorites"
  - 1.4 在单集列表项的长按 / 上下文菜单中点击"Add to favorites"或"Remove from favorites"
  - 1.5 在支持滑动的单集列表上对该单集执行收藏方向的滑动
- 2. 系统判定该单集当前是否已收藏：
  - 2.1 若当前未收藏，则将其标记为已收藏
  - 2.2 若当前已收藏，则取消其已收藏标记
- 3. 操作立即生效，收藏状态写入并持久化
- 4. 星标按钮 / 列表项收藏角标立即反映新状态：
  - 4.1 已收藏时，播放页星标变为实心，单集列表项显示收藏角标
  - 4.2 未收藏时，播放页星标变为空心，单集列表项不显示收藏角标
- 5. Favorites 页面随后同步反映该单集的加入或移出（详见场景三、场景四）

## 场景二

### 场景概述

用户在多选模式下批量收藏或批量取消收藏多条单集

### 场景逻辑步骤

- 1. 用户在任一支持多选的单集列表中进入多选模式，选中两条及以上单集
- 2. 用户点击批量动作中的"Add to favorites"或"Remove from favorites"
- 3. 系统对选中单集批量写入或取消收藏标记：
  - 3.1 批量收藏时，仅对其中尚未收藏的单集生效
  - 3.2 批量取消收藏时，仅对其中已收藏的单集生效
- 4. 当实际发生状态变化的单集数量大于 1 时，弹出数量浮动提示，显示本次加入或移出的单集数量；数量为 1 时不弹出
- 5. 各单集的星标 / 角标与 Favorites 页面同步刷新

## 场景三

### 场景概述

单集处于已收藏状态时的运行时呈现

### 场景逻辑步骤

- 1. 某单集已被标记为收藏
- 2. 该单集出现在 Favorites 页面的列表中（可通过底部"更多" → "Favorites" 查看）
- 3. 该单集在音频/视频播放页的工具栏星标显示为实心星
- 4. 该单集在所有单集列表项上显示收藏角标
- 5. 该单集的选项菜单中"Add to favorites"项隐藏，"Remove from favorites"项可见

## 场景四

### 场景概述

单集处于未收藏状态时的运行时呈现

### 场景逻辑步骤

- 1. 某单集未被标记为收藏
- 2. 该单集不出现在 Favorites 页面
- 3. 该单集在音频/视频播放页的工具栏星标显示为空心星
- 4. 该单集在所有单集列表项上不显示收藏角标
- 5. 该单集的选项菜单中"Add to favorites"项可见，"Remove from favorites"项隐藏

## 场景五

### 场景概述

"收藏切换"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下界面与数据：Favorites 页面的列表内容；单集列表项的收藏角标；音频/视频播放页工具栏星标的形态；单集选项菜单中收藏项的可见性
- 2. **不作用于** 以下界面：播放队列、Inbox、下载、播放历史 — 这些列表各自依据独立标记渲染，收藏切换不影响其项数、排序与存在
- 3. **无隐式触发**：收藏状态仅由用户主动操作改变；订阅源刷新、同步服务等后台流程不写入收藏标记

> [偏差] 需求原文仅提及"单集详情或播放页点击星标"两类入口；为完整覆盖代码已发现的全部入口，本 spec 将单集列表上下文菜单、多选批量、列表滑动一并纳入（见场景一步骤 1 与场景二）。需求未提及批量数量浮动提示；代码在批量且数量大于 1 时弹出（见场景二步骤 4）。

# Favorites 页面SPEC

用户通过底部"More"溢出菜单（或自定义后的主底部导航）进入 Favorites 页面，查看已收藏的单集列表；无收藏时显示空态提示；列表项可进入单集详情，并支持滑动移出收藏、多选批量操作、搜索与刷新。

## 场景一

### 场景概述

用户从底部"More"进入 Favorites 页面，页面标题与导航行为符合预期

### 场景逻辑步骤

- 1. 用户在应用底部导航点击"More"，弹出溢出菜单
- 2. 用户在溢出菜单中点击"Favorites"
- 3. 进入 Favorites 页面，页面顶部工具栏标题显示为"Favorites"，并显示返回箭头

无任何收藏时，Favorites 页面显示空态

- 2. 列表区域显示空态：星形图标、标题"No favorites"、说明文案"Mark episodes as favorites by tapping the star icon."

存在收藏时，Favorites 页面展示收藏单集列表，并可进入单集详情

### 场景逻辑步骤

- 2. 列表展示全部已收藏单集，每项显示封面、播客名、单集标题、发布日期、时长、播放/下载等状态，以及收藏角标

- 2. 用户选中两条及以上单集，底部出现批量动作栏
- 3. 批量动作中至少包含"移出收藏"，也可包含加入队列、下载、标记已播放等通用单集动作

## 场景六

### 场景概述

"Favorites 页面"的作用范围与不作用范围

### 场景逻辑步骤

- 2. **不作用于** 以下界面：播放队列、Inbox、下载、播放历史 — 这些列表各自依据独立标记渲染，不混入收藏数据；底部导航的 Inbox 角标仅反映新单集数量，与收藏无关

> [偏差] 需求原文仅提及"通过 More 菜单进入"与"空态/有数据"两种情形；为完整覆盖代码已发现的全部入口与能力，本 spec 将自定义主底部导航入口、滑动移出收藏、多选批量、搜索与刷新一并纳入。需求未提及排序；Favorites 复用全局"全部单集"排序，页面本身无独立排序入口（见场景三步骤 3）。需求未提及滑动移出收藏与多选批量；代码默认提供（见场景四、场景五）。

# 播放单集SPEC

用户在单集上发起播放（已下载为 Play、未下载偏好流播为 Stream），单集进入播放状态；应用底部出现迷你播放条，可展开为完整播放页；播放页展示封面、播客名、发布日期、单集标题及播放控制；底部导航状态保持。

## 场景一

### 场景概述

用户在单集列表项上点击次要动作按钮，依据单集状态进入播放或流播

### 场景逻辑步骤

- 1. 用户在任一单集列表（Episodes、Inbox、Queue、Favorites、播客详情等）看到某条单集封面右侧的圆形次要动作按钮
- 2. 系统依据该单集当前状态自动决定按钮形态：
  - 2.1 单集已下载 → 按钮显示为"Play"
  - 2.2 单集未下载、未在下载中，且用户偏好流播 → 按钮显示为"Stream"
  - 2.3 单集未下载且偏好下载 → 按钮显示为"Download"（点击后下载，不直接播放）
  - 2.4 单集下载中 → 按钮显示为"Cancel download"
  - 2.5 单集正在播放 → 按钮显示为"Pause"
- 3. 用户点击"Play"或"Stream"按钮：
  - 3.1 点击"Play"时，先校验已下载文件是否存在；若文件丢失，回退为未下载状态并弹出"File not found"提示，不进入播放
  - 3.2 文件存在或点击"Stream"时，启动播放服务装载该单集媒体，进入播放状态
- 4. 进入播放状态后，主界面底部出现迷你播放条

## 场景二

### 场景概述

单集进入播放状态后，底部出现迷你播放条，并可展开为完整播放页

### 场景逻辑步骤

- 1. 某单集进入播放状态后，主界面底部导航之上出现迷你播放条
- 2. 迷你播放条展示：单集封面、单集标题、播客名、播放进度条、播放/暂停按钮
- 3. 播放/暂停按钮反映当前播放状态：正在播放时显示"暂停"，暂停时显示"播放"
- 4. 用户点击迷你播放条的播放/暂停按钮，在播放与暂停之间切换
- 5. 用户点击迷你播放条主体（封面/标题区域），底部播放区域展开为完整音频播放页
- 6. 底部导航仍保持可见，当前选中页不变

## 场景三

### 场景概述

完整音频播放页展示当前播放单集信息

### 场景逻辑步骤

- 1. 用户从迷你播放条展开进入完整音频播放页
- 2. 播放页展示当前播放单集的：封面、播客名、发布日期（与播客名同行显示）、单集标题
- 3. 播放页提供播放/暂停按钮，并可见后退、快进、跳过、播放速度、进度条、播放时间等控制（各项控制的详细行为见 `player-controls-SPEC.md`）
- 4. 单集标题过长时，用户点击标题可滚动查看完整标题
- 5. 用户点击播放页顶部折叠按钮，播放页收回为迷你播放条，播放状态不中断

## 场景四

### 场景概述

未下载单集在移动网络下流播时，需用户确认

### 场景逻辑步骤

- 1. 用户对一条未下载单集点击"Stream"（或队列连续播放触发流播下一集）
- 2. 系统检测到当前为移动网络，且设置中未允许移动网络流播
- 3. 系统不立即开始流播，而是发出流播确认提示
- 4. 用户确认后，本次允许流播，开始播放；用户拒绝则不播放
- 5. 该"本次允许"仅对当次流播生效，不改变设置中的移动网络流播开关

## 场景五

### 场景概述

视频单集播放打开独立的视频播放界面

### 场景逻辑步骤

- 1. 用户对一条视频单集点击"Play"或"Stream"
- 2. 系统启动播放服务，并打开独立的视频播放界面（而非底部音频播放页）
- 3. 视频播放界面承载视频画面与播放控制
- 4. 返回后回到原页面，播放可在后台继续（音频）

## 场景六

### 场景概述

"播放单集"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下界面与数据：迷你播放条与完整音频播放页的当前播放单集信息；单集列表项的次要动作按钮形态（依据播放状态切换）；底部导航底部边距（为迷你播放条让出空间）
- 2. **不作用于** 以下界面：底部导航的当前选中页（播放不改变选中页）；播放队列的顺序（除非连续播放消耗队列，属 REQ-017 范畴）
- 3. **隐式触发**：当前单集播放结束且开启连续播放时，自动装载并播放队列中的下一集；移动网络流播需确认（见场景四）；系统媒体按钮、通知栏播放控制可触发同一播放服务的播放/暂停

> [偏差] 需求原文表述为"点击 Stream/Play"；代码中列表项次要动作是动态的（Play/Stream/Download/Cancel/Pause），仅"已下载"与"未下载且偏好流播"两种形态对应 REQ 所述（见场景一步骤 2）。需求提到"播客语音单集"（音频）；代码额外支持视频单集，走独立视频播放界面（见场景五）。需求未提及移动网络流播确认、文件缺失回退、队列连续播放；代码均提供（见场景一、场景四、场景六）。播放页各项控制的详细行为另见 `player-controls-SPEC.md`。

# 播放页控制SPEC

用户在完整音频播放页可控制播放/暂停、后退、快进、跳过下一集、播放速度、进度条定位、睡眠定时器、收藏、分享及其它入口；操作后控件状态与时间文本立即反馈，进度或时间合理变化。

## 场景一

### 场景概述

用户切换播放与暂停

### 场景逻辑步骤

- 1. 用户在播放页点击播放/暂停按钮，或点击封面区域（等价触发）
- 2. 系统依据当前状态切换：
  - 2.1 当前正在播放 → 暂停
  - 2.2 当前未播放 → 从当前位置恢复播放
- 3. 播放/暂停按钮的形态立即更新为对应状态
- 4. 进度条与时间文本随之停止或继续刷新

## 场景二

### 场景概述

用户后退或快进指定秒数

### 场景逻辑步骤

- 1. 用户点击"后退"按钮，从当前位置向后退预设秒数
- 2. 用户点击"快进"按钮，从当前位置向前快进预设秒数
- 3. 按钮旁显示当前生效的后退/快进秒数
- 4. 长按"后退"或"快进"按钮弹出秒数设置，用户可修改预设秒数

### 场景概述

用户跳过当前单集，播放队列中的下一集

### 场景逻辑步骤

- 1. 用户点击"跳过"按钮
- 2. 系统装载并播放队列中当前单集之后的下一集
- 3. 播放页的封面、播客名、日期、标题、进度、时间文本同步刷新为新单集内容

用户调整播放速度

用户拖动进度条定位到指定位置

  - 2.2 否则浮层显示目标位置的时间

- 4. 当前时间文本与进度条立即更新；继续播放从新位置开始

## 场景六

### 场景概述

用户切换时间文本的显示方式

用户设置或关闭睡眠定时器

用户在播放页收藏当前单集或分享当前单集

## 场景九

用户使用播放页工具栏的"更多"入口

# Playback historySPEC

用户通过底部"More"溢出菜单（或自定义后的主底部导航）进入 Playback history 页面，查看已播放完成的单集；列表按完成时间倒序排列，可进入单集详情，支持滑动删除单条、清空全部历史与多选批量操作。

## 场景一

### 场景概述

用户从底部"More"进入 Playback history 页面

### 场景逻辑步骤

- 1. 用户在应用底部导航点击"More"，弹出溢出菜单
- 2. 用户在溢出菜单中点击"Playback history"
- 3. 进入播放历史页面，页面顶部工具栏标题显示为"Playback history"，并显示返回箭头
- 4. 若用户此前把 History 调入了主底部导航，也可直接点击底部"History"进入，行为一致
- 5. 点击返回箭头可回到进入前的页面

## 场景二

### 场景概述

无任何播放历史时，页面显示空态

### 场景逻辑步骤

- 1. 用户进入 Playback history 页面，且没有任何已播放完成的单集
- 2. 列表区域显示空态：历史图标、标题"No history"、说明文案"After you listen to an episode, it will appear here."
- 3. 工具栏的"Clear history"按钮在列表为空时隐藏
- 4. 当用户播放完成任一单集后，空态消失，列表出现该单集

## 场景三

### 场景概述

存在播放历史时，页面展示历史单集列表，并可进入单集详情

### 场景逻辑步骤

- 1. 用户进入 Playback history 页面，且至少有一条单集被播放完成
- 2. 列表展示全部已播放完成的单集，按完成时间倒序排列（最近完成的在最前）
- 3. 每项显示封面、播客名、单集标题、发布日期、时长、播放/下载等状态
- 4. 列表支持向下滚动加载更多
- 5. 用户点击某个列表项，进入该单集的详情/播放上下文
- 6. 当用户播放完成新单集或从历史删除单集时，本列表实时刷新，无需手动刷新

## 场景四

### 场景概述

用户在历史列表上滑动单集以移出历史

### 场景逻辑步骤

- 1. 用户在 Playback history 页面对某条单集执行移出历史方向的滑动
- 2. 该单集的播放完成记录被清除，从历史列表消失
- 3. 该单集仍保留在播客详情与 Episodes 中，不删除本体
- 4. 列表数量与顺序同步更新

## 场景五

### 场景概述

用户清空全部播放历史

### 场景逻辑步骤

- 1. 用户在 Playback history 页面点击工具栏的"Clear history"按钮（列表为空时该按钮隐藏）
- 2. 弹出确认对话框，提示"This will clear the entire playback history. Are you sure you want to proceed?"
- 3. 用户在确认对话框中：
  - 3.1 点击"取消"，关闭对话框，历史保持不变
  - 3.2 点击"确认"，清空全部播放历史
- 4. 清空完成后，列表变为空态，"Clear history"按钮隐藏
- 5. 清空操作仅清除播放完成记录，不删除任何单集本体，也不影响播放队列、收藏、下载

## 场景六

### 场景概述

用户在 Playback history 页面进入多选模式，对历史单集进行批量操作

### 场景逻辑步骤

- 1. 用户在 Playback history 页面长按某条单集进入多选模式
- 2. 用户选中两条及以上单集，底部出现批量动作栏
- 3. 批量动作可包含加入队列、下载、标记已播放/未播放、删除等通用单集动作（标记未播放会间接把单集移出历史）
- 4. 用户点击某个批量动作，对选中单集执行，并视情况弹出数量浮动提示
- 5. 用户可退出多选模式，列表恢复正常浏览

## 场景七

### 场景概述

"Playback history"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下界面与数据：Playback history 页面的列表内容（仅已播放完成的单集）；历史列表项的已播放态显示；工具栏"Clear history"按钮的可见性
- 2. **不作用于** 以下界面与数据：播放队列、Inbox、收藏、下载 — 这些列表各自依据独立标记渲染，清空历史或移出历史不影响这些列表的存在与状态
- 3. **隐式触发**：单集播放完成时自动写入历史并刷新列表；从历史删除单条或清空全部时实时刷新

> [偏差] 需求原文仅提及"通过 More 菜单进入"、"播放过的单集出现"、"可进入单集详情"、"支持清空历史"；为完整覆盖代码已发现的全部入口与能力，本 spec 将自定义主底部导航入口、滑动删除单条、多选批量一并纳入（见场景一、四、六）。需求未提及排序；历史固定按完成时间倒序，无独立排序入口（见场景三步骤 2）。需求未提及历史写入时机；代码在单集播放完成时由播放服务写入（见场景二步骤 4 与场景七）。

## 场景六

用户一键删除全部已播放的下载

### 场景逻辑步骤

- 1. 用户在 Downloads 页面工具栏溢出菜单点击"Delete played"
- 2. 弹出确认对话框，提示"Please confirm that you want to delete all played downloads."
- 3. 用户在确认对话框中：
  - 3.1 点击"取消"，关闭对话框，下载保持不变
  - 3.2 点击"确认"，删除全部已播放的下载
- 4. 删除完成后，Downloads 列表仅保留未播放的下载；未被播放过的下载不受影响

## 场景七

### 场景概述

用户从 Downloads 页面打开下载日志

### 场景逻辑步骤

- 1. 用户在 Downloads 页面点击工具栏的下载日志入口
- 2. 弹出下载日志（详见 `download-log-SPEC.md`），展示下载记录与失败详情
- 3. 关闭下载日志后回到 Downloads 页面
- 4. 在某些情形下（如刚发生下载失败时），进入 Downloads 页面会自动弹出下载日志

## 场景八

### 场景概述

"Downloads 页面"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下界面与数据：Downloads 页面的列表内容（正在下载 + 已下载单集）；已下载单集的次要动作形态（播放或删除）；列表项的下载进度与已下载角标
- 2. **不作用于** 以下界面与数据：播放队列、Inbox、收藏、播放历史 — 这些列表各自依据独立标记渲染，下载状态变化不混入这些列表
- 3. **隐式触发**：下载进度/状态变化、单集状态变化、播放状态变化、订阅源刷新进行中等都会触发下载页面刷新

> [偏差] 需求原文仅提及"通过 More 菜单进入"、"查看下载中或已下载单集"、"已下载单集可播放或删除下载"、"删除下载后列表更新"；为完整覆盖代码已发现的全部入口与能力，本 spec 将自定义主底部导航入口、多选批量、一键删除已播放、下载日志入口、排序与搜索一并纳入（见场景一、五、六、七、三步骤 5）。需求未提及"正在下载"项的展示；代码同时展示正在下载与已下载（见场景三步骤 2）。需求未提及已下载项次要动作在"播放/删除"间依据设置切换；代码提供（见场景四步骤 2）。下载日志的详细行为另见 `download-log-SPEC.md`。

<!-- REQ-029: download-log -->
# Download logSPEC

用户可在 Downloads 页面打开下载日志，查看每次下载（订阅源或单集媒体）的完成或失败记录；失败记录附带可理解的错误原因，可查看详情与重试；支持清空日志；错误状态不会导致应用崩溃。

## 场景一

### 场景概述

用户从 Downloads 页面打开下载日志

### 场景逻辑步骤

- 1. 用户进入 Downloads 页面
- 2. 用户点击页面工具栏的下载日志入口
- 3. 弹出下载日志（底部弹层），按时间倒序列出最近的下载记录
- 4. 每条记录显示：下载类型（订阅源 / 单集媒体）、完成时间的相对描述、标题、成功或失败图标
- 5. 用户可上下滚动浏览历史记录
- 6. 用户可通过下滑或返回关闭下载日志，回到 Downloads 页面

## 场景二

### 场景概述

下载日志为空时的展示

### 场景逻辑步骤

- 1. 用户打开下载日志，且没有任何下载记录
- 2. 日志区域显示空态：下载图标、标题"No download log"、说明文案"Download logs will appear here when available."
- 3. 当发生一次下载（成功或失败）后，空态消失，日志出现该记录

## 场景三

### 场景概述

下载成功的记录展示

### 场景逻辑步骤

- 1. 用户打开下载日志，查看某条下载成功的记录
- 2. 该记录显示成功图标与"successful"描述
- 3. 该记录不显示错误原因，也不显示重试按钮
- 4. 该记录的标题为对应订阅源或单集的标题

## 场景四

### 场景概述

下载失败的记录展示与错误原因

### 场景逻辑步骤

- 1. 用户打开下载日志，查看某条下载失败的记录
- 2. 该记录显示失败图标（部分重复解析类错误显示信息图标）
- 3. 该记录显示可理解的错误原因文案（由错误类型映射而来），并提示"可点击查看详情"
- 4. 若该标题为空，则显示"Unknown title"
- 5. 错误状态仅作为记录展示，不会导致应用崩溃

## 场景五

### 场景概述

用户查看失败记录的详情

### 场景逻辑步骤

- 1. 用户在下载日志中点击某条失败记录
- 2. 弹出详情对话框，展示：
  - 2.1 可读的状态描述
  - 2.2 技术原因（错误的技术细节）
  - 2.3 文件来源地址
- 3. 用户关闭详情对话框后回到下载日志

## 场景六

### 场景概述

用户对失败的下载发起重试

### 场景逻辑步骤

- 1. 用户在下载日志中查看某条失败记录
- 2. 若该记录之后没有更新的成功记录，则显示重试按钮
- 3. 用户点击重试按钮：
  - 3.1 若失败对象是订阅源，触发该订阅源刷新
  - 3.2 若失败对象是单集媒体，触发该单集重新下载，并弹出"Episode is being downloaded"提示
- 4. 重试后，新的下载记录会写入日志并刷新
- 5. 若该记录之后已有更新的成功记录，则不显示重试按钮（无需重试）

## 场景七

### 场景概述

用户清空下载日志

### 场景逻辑步骤

- 1. 用户在下载日志顶部工具栏点击"Clear history"按钮
- 2. 清空全部下载日志记录
- 3. 日志区域变为空态
- 4. 清空操作不影响已下载的单集本体、订阅、播放队列或播放历史

## 场景八

### 场景概述

下载失败后自动弹出下载日志

### 场景逻辑步骤

- 1. 用户触发一次下载，该下载失败（或在某些情形下用户从失败提示进入 Downloads）
- 2. 当用户随后进入 Downloads 页面时，下载日志自动弹出
- 3. 用户可直接查看刚发生的失败记录与原因
- 4. 用户也可关闭弹层，后续仍可从 Downloads 工具栏手动打开

## 场景九

### 场景概述

"Download log"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下界面与数据：下载日志弹层的记录内容（仅下载记录）；失败记录的错误原因、详情与重试
- 2. **不作用于** 以下数据：播放历史、收藏、播放队列 — 这些数据不混入下载日志；清空下载日志不影响这些数据
- 3. **隐式触发**：每次下载完成或失败时由下载服务写入一条记录并刷新日志；清空日志、删除单集/订阅源时日志相应更新

> [偏差] 需求原文仅提及"查看下载日志和失败详情"、"失败记录提供可理解的错误信息"、"错误状态不导致崩溃"；为完整覆盖代码已发现的全部入口与能力，本 spec 将重试按钮、清空日志、自动弹出、空态展示一并纳入（见场景六、七、八、二）。需求未提及下载日志入口的具体位置；代码入口在 Downloads 页面工具栏，并在特定情形下自动弹出（见场景一、八）。

# 应用内搜索SPEC

用户在任一主页顶部工具栏点击搜索图标进入搜索页，输入关键词后实时（带防抖）刷新本地匹配的播客与单集结果；点击播客结果进入播客详情，点击单集结果进入单集详情；并提供在线搜索入口与结果项的批量操作。

## 场景一

### 场景概述

用户从主页顶部进入搜索页

### 场景逻辑步骤

- 1. 用户在任一主页（Home、Queue、Inbox、Episodes、Subscriptions、Favorites、Playback history、Downloads）顶部工具栏点击搜索图标
- 2. 进入搜索页，页面标题为"Search"，并显示返回箭头
- 3. 搜索框自动展开并获得焦点，弹出输入键盘
- 4. 搜索框提示文案为"Search"
- 5. 用户点击返回箭头或收起搜索框可回到进入前的页面

## 场景二

### 场景概述

用户输入关键词，结果按关键词实时刷新

### 场景逻辑步骤

- 1. 用户在搜索框输入关键词
- 2. 系统对输入做防抖处理（连续快速输入时不立即查询，停顿约半秒到一秒后再查询；若输入以空格结尾或停顿超过阈值则立即查询）
- 3. 查询在本地已订阅播客与已知单集中匹配，结果分两部分展示：
  - 3.1 顶部横向列表展示匹配的播客
  - 3.2 下方纵向列表展示匹配的单集
- 4. 用户继续输入或修改关键词，结果随之刷新
- 5. 用户清空搜索框，结果区域回到"请输入要搜索的内容"的初始提示

## 场景三

### 场景概述

无匹配结果时的展示

### 场景逻辑步骤

- 1. 用户输入一个关键词并完成查询
- 2. 若本地没有任何播客或单集匹配该关键词
- 3. 结果区域显示"No results were found for '<关键词>'"
- 4. 用户可修改关键词重新搜索，或使用"在线搜索"入口转到在线搜索

## 场景四

### 场景概述

用户点击播客结果进入播客详情

### 场景逻辑步骤

- 1. 用户在搜索结果的顶部横向播客列表中看到匹配的播客
- 2. 用户点击某个播客结果
- 3. 进入该播客的详情页（单集列表）
- 4. 返回后回到搜索结果，结果与搜索框内容保持

## 场景五

### 场景概述

用户点击单集结果进入单集详情

### 场景逻辑步骤

- 1. 用户在搜索结果的单集列表中看到匹配的单集
- 2. 每条单集结果展示封面、播客名、标题、日期、时长、播放/下载状态等
- 3. 用户点击某个单集结果
- 4. 进入该单集的详情/播放上下文
- 5. 返回后回到搜索结果，结果与搜索框内容保持

## 场景六

### 场景概述

用户使用在线搜索

### 场景逻辑步骤

- 1. 用户在搜索页底部看到"Search online"入口（在指定播客内搜索或带单集级过滤时不显示）
- 2. 用户点击"Search online"
- 3. 系统依据关键词形态分流：
  - 3.1 若关键词是一个网址，直接进入"添加订阅源"的在线预览
  - 3.2 否则进入在线搜索页，联合多个在线播客源搜索
- 4. 在线搜索结果可与本地搜索结果区分，命中后可进一步订阅或查看

## 场景七

### 场景概述

用户对单集搜索结果进行上下文操作或批量操作

### 场景逻辑步骤

- 1. 用户在单集搜索结果上长按某条结果，打开上下文菜单，可执行加入队列、下载、收藏、分享、标记已播放等单集操作
- 2. 用户也可进入多选模式，选中两条及以上结果，从底部批量动作栏执行批量操作
- 3. 操作完成后结果列表相应刷新

## 场景八

### 场景概述

"应用内搜索"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下界面与数据：搜索页的播客结果列表与单集结果列表（基于本地已订阅播客与已知单集）；结果项的状态角标（依据各自的播放/下载/收藏/队列标记）
- 2. **不作用于** 以下数据：未订阅的在线播客（由"在线搜索"单独处理，不混入本地结果）；播放队列/收藏/历史的顺序与成员（搜索仅读取这些标记用于过滤，不修改它们）
- 3. **隐式触发**：订阅源增删、单集状态变化、下载进度变化、播放位置变化都会触发搜索结果相应刷新

> [偏差] 需求原文仅提及"顶部搜索入口"、"输入关键词显示匹配结果"、"点击结果进入详情"；为完整覆盖代码已发现的全部入口与能力，本 spec 将多主页搜索入口、防抖实时刷新、在线搜索、空态、结果项上下文与批量操作一并纳入（见场景一、二、三、六、七）。需求未提及"播客结果"与"单集结果"分两段展示；代码同时返回并分区展示（见场景二步骤 3、场景四、场景五）。播客内搜索（限定在某个播客内的单集搜索）另见 REQ-031，复用同一搜索页。

# 播客内搜索SPEC

用户在某个订阅播客的详情页中，可以搜索该播客内的单集；搜索范围被默认限制在该播客，搜索结果只包含标题或描述命中关键词的单集。

## 场景一

### 场景概述

用户在订阅播客详情页中打开搜索入口，进入"仅限当前播客"的搜索结果页

### 场景逻辑步骤

- 1. 用户从"订阅页"或其它入口进入某个已订阅播客的"播客详情页"
- 2. 用户点击"播客详情页"顶部工具栏的"搜索"图标
- 3. 应用以该播客的标识和播客名为参数打开"搜索结果页"
- 4. "搜索结果页"加载时：
  - 4.1 搜索框获得输入焦点并弹出输入键盘
  - 4.2 搜索框下方显示一个代表"当前搜索范围"的播客名标签（可关闭）
  - 4.3 在用户输入关键词之前，结果区显示"请输入要搜索的内容"空态
  - 4.4 隐藏"在线搜索更多播客"入口（播客内搜索不引导用户联网发现新播客）

> [偏差] 需求未提及"播客名标签"和"在线搜索入口的隐藏"；代码默认提供上述两者，详见场景五。

## 场景二

### 场景概述

用户在"播客内搜索结果页"输入关键词，结果只来自当前播客

### 场景逻辑步骤

- 1. 用户处于"播客内搜索结果页"（场景一第 3 步之后）
- 2. 用户在搜索框中输入单集关键词
- 3. 输入过程中应用按一定节流策略异步发起查询；当输入以空格结尾、或停顿超过节流时长时立即查询
- 4. 查询时按如下规则匹配单集：
  - 4.1 关键词按空白拆分为多个查询词，单集必须同时命中全部查询词
  - 4.2 命中字段仅限单集"标题"和"描述"两个文本字段；其它字段（如作者、单集正文）不参与匹配
  - 4.3 命中规则为子串包含（不区分大小写由数据库默认排序规则决定）
- 5. 命中单集按"发布时间倒序"排列，结果数量上限为 300 条
- 6. 结果区刷新为命中单集列表；若没有任何单集命中，结果区显示"未找到与 `<关键词>` 相关的结果"空态
- 7. 用户也可显式提交（键盘确认键）触发一次带进度条的强制刷新

> [偏差] 需求未提及命中字段范围；代码仅匹配标题与描述。需求未提及结果上限；代码强制 300 条且按发布时间倒序。

## 场景三

### 场景概述

用户点击"播客内搜索结果页"中的某个结果项，进入对应单集详情

### 场景逻辑步骤

- 1. 用户在"播客内搜索结果页"的结果列表中点击某个单集
- 2. 应用打开该单集的"单集详情页"，详情内容与点击的 result 一一对应
- 3. 用户从"单集详情页"返回后，仍回到"播客内搜索结果页"，搜索框内容与结果列表保持不变

## 场景四

### 场景概述

"播客内搜索结果页"在以下后台事件发生时自动重新查询当前播客范围

### 场景逻辑步骤

- 1. 用户停留在"播客内搜索结果页"期间，应用持续监听后台事件
- 2. 当发生"订阅源列表变更"或"当前结果所属单集发生变更且为未读状态变更"或"播放状态变化"时，应用自动重新执行当前关键词在当前播客范围内的查询并刷新结果列表
- 3. 当发生"下载进度更新"或"播放进度更新"事件时，应用不重新查询，仅对已存在于结果列表中的相应单集项刷新其状态徽标与进度
- 4. 上述自动刷新始终保持在"当前播客范围内"，不会因事件触发而把范围扩大到全部订阅源

## 场景五

### 场景概述

用户通过关闭"播客名标签"主动把搜索范围扩大到全部订阅源

### 场景逻辑步骤

- 1. 用户处于"播客内搜索结果页"，搜索框下方仍显示代表当前范围的"播客名标签"
- 2. 用户点击该标签的关闭按钮
- 3. 应用把当前搜索范围标记为"全部订阅源"，移除该"播客名标签"
- 4. 应用按当前关键词立即重新查询，结果区刷新为全部订阅源范围内的命中单集（仍受同样的命中字段、排序与上限规则约束）
- 5. 之后用户的搜索行为等同于"全应用搜索"

> [偏差] 需求描述搜索结果"限制在该播客"，但代码提供了用户可主动解除该限制的入口；本场景如实记录该可解除行为，规格默认行为仍以"限制在当前播客"为准。

## 场景六

### 场景概述

"播客内搜索"功能的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下入口: 已订阅播客的"播客详情页"顶部工具栏"搜索"图标
- 2. **不作用于** 以下位置: "未订阅预览态"的播客详情页（不显示搜索图标，亦不响应）；"单集详情页" / "播放页" / "订阅页" / "首页" / "队列页" / "收件箱页" / "收藏页" / "下载页" / "历史页" / "全应用搜索结果页" 中的搜索入口均属于"全应用搜索"，不限定到某个播客；"添加播客页"中的搜索为在线发现新播客，与本功能无关
- 3. **不作用于** 以下数据范围: 单集作者、单集正文、章节标题等字段均不参与"播客内搜索"匹配

# 播客设置SPEC

每个订阅播客拥有一个独立的播客设置页，用户可在此页修改该播客的展示、播放、自动化与高级类偏好；任何修改在用户确认后立即写入应用的本地存储并同步刷新相关页面，返回播客详情页后界面状态与新设置保持一致。

## 场景一

### 场景概述

用户从播客详情页打开该播客的设置页

### 场景逻辑步骤

- 1. 用户从"订阅页"或其它入口进入某个已订阅播客的"播客详情页"
- 2. 用户点击"播客详情页"头部的"设置"按钮
- 3. 应用以该播客标识为参数打开"播客设置页"
- 4. "播客设置页"加载时：
  - 4.1 顶部显示返回按钮与页面标题，副标题显示当前播客的标题
  - 4.2 在该播客的偏好加载完成之前，整页设置项保持隐藏，避免显示半加载状态
  - 4.3 偏好加载完成后，按"展示 / 播放 / 自动化 / 高级"四个分组依次展示该播客的所有可选项

## 场景二

### 场景概述

用户在"播客设置页"修改一个开关类设置（保持更新、单集通知）

### 场景逻辑步骤

- 1. 用户在"播客设置页"找到一个开关类设置项（如"保持更新"、"单集通知"）
- 2. 用户切换开关
- 3. 应用按如下分支处理：
  - 3.1 "保持更新"开关：立即写入新状态，开关 UI 同步刷新；"单集通知"项的可用性跟随"保持更新"的状态（关闭"保持更新"时"单集通知"不可用）
  - 3.2 "单集通知"开关从关闭切换到开启：若系统要求通知权限且当前未授权，应用先发起通知权限申请；用户授权后才把开关置为开启并写入；用户拒绝时开关保持原关闭状态，不写入
  - 3.3 "单集通知"开关从开启切换到关闭：直接写入新状态
- 4. 写入成功后，应用发出"订阅源变更"事件，相关页面据此刷新
- 5. 用户从"播客设置页"返回"播客详情页"后，相关行为按新设置生效

## 场景三

### 场景概述

用户在"播客设置页"修改一个列表选择类设置（自动删除、新单集动作、自动下载、音量适配）

### 场景逻辑步骤

- 1. 用户点击某个列表选择类设置项（如"自动删除"、"新单集动作"、"自动下载"、"音量适配"）
- 2. 应用按如下分支处理：
  - 2.1 当点击"新单集动作"且该播客当前已开启自动下载、且不是本地文件夹订阅时：应用弹出消息提示"自动下载开启时无法直接切换新单集动作"，本次不弹出选项列表，也不写入
  - 2.2 其它情况：弹出选项列表，并标记当前生效值为选中态
- 3. 用户在选项列表中选择一个值（场景二的 2.1 阻塞分支除外）
- 4. 应用把新值写入本地存储，刷新该项的副标题为新值对应的文案
- 5. 写入成功后，应用发出"订阅源变更"事件
- 6. 用户返回"播客详情页"后，相关行为按新值生效

## 场景四

### 场景概述

用户在"播客设置页"修改一个弹窗类设置（播放速度、自动跳过片头/片尾、认证、标签、单集筛选、重命名、编辑源地址、重连本地文件夹）

### 场景逻辑步骤

- 1. 用户点击某个弹窗类设置项
- 2. 应用弹出对应输入或确认弹窗，弹窗初始值显示该播客当前值
- 3. 用户在弹窗中输入新值后确认；用户也可取消关闭弹窗
- 4. 当用户确认时，应用按如下分支处理：
  - 4.1 "认证"项：写入新用户名与密码后，应用立即按新凭据对该播客发起一次刷新
  - 4.2 "标签"项：打开"标签管理"子页（具体行为见 REQ-033 标签与分组）
  - 4.3 其它项：直接把新值写入本地存储
- 5. 写入成功后，应用发出"订阅源变更"事件，相关页面同步刷新

## 场景五

### 场景概述

"本地文件夹订阅"在"播客设置页"中看到的可选项集合与普通订阅不同

### 场景逻辑步骤

- 1. 用户打开一个"本地文件夹订阅"的"播客设置页"
- 2. 应用在偏好加载完成后：
  - 2.1 隐藏"认证"项
  - 2.2 隐藏"自动下载"项
  - 2.3 隐藏"单集筛选"项
  - 2.4 隐藏"编辑源地址"项
  - 2.5 显示"重连本地文件夹"项（仅本地订阅可见）
- 3. 用户修改其它仍然可见的设置项时，行为与场景二 / 三 / 四一致
- 4. 当用户点击"重连本地文件夹"项时，应用弹出确认弹窗；用户确认后进入系统文件夹选择流程以重新关联本地文件夹

> [偏差] 需求未区分"本地文件夹订阅"与普通订阅；代码对本地订阅隐藏了认证 / 自动下载 / 单集筛选 / 编辑源地址，并新增"重连本地文件夹"项。

## 场景六

### 场景概述

"播客设置"功能的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下入口: 已订阅播客的"播客详情页"头部"设置"按钮
- 2. **不作用于** 以下位置: "全局应用设置"页（属于全局偏好域，与播客级偏好互相独立）；"播客详情页"头部"详情信息"按钮（打开的是只读信息页）；"播客详情页"头部"筛选"按钮（仅修改单集筛选，不覆盖本 REQ 全部设置项）；"添加播客"流程中的预览态播客（未订阅，不进入播客设置页）
- 3. **不作用于** 以下数据范围: 本规格只覆盖播客级偏好；"主题、播放快进秒数、全局下载间隔"等全局设置归属于"全局应用设置"，不在本规格内

# 标签与分组SPEC

用户可给订阅播客添加一个或多个自定义标签，应用会按标签把订阅聚合成"标签分组"，并在订阅页（顶部标签列表）或抽屉式导航（可折叠分组）中展示与切换这些分组；标签保存成功后即时反映到所有展示面。

## 场景一

### 场景概述

用户在"播客设置页"打开"编辑标签"对话框，添加或删除该播客的标签

### 场景逻辑步骤

- 1. 用户进入某个已订阅播客的"播客设置页"
- 2. 用户点击"标签"项
- 3. 应用弹出"编辑标签"对话框，加载时：
  - 3.1 已有标签区按 2 列网格展示该播客当前的所有自定义标签（每个标签显示为一个带关闭按钮的 chip）；不显示"全部"伪标签
  - 3.2 下方显示一个"添加新标签"输入框，输入框带有自动补全：候选项来自当前应用内已经存在的全部自定义标签
  - 3.3 当底部导航布局启用时，对话框不显示"显示在根目录"勾选框；否则按该播客当前是否在根目录决定勾选状态
- 4. 用户在对话框中：
  - 4.1 添加新标签：在输入框输入标签名后点击确认图标追加；空字符串、与已有标签重复、或等于系统保留名"未分组"的输入被忽略，输入框保持焦点等待重新输入
  - 4.2 删除已有标签：点击该 chip 上的关闭按钮，立即从展示区移除
  - 4.3 切换"显示在根目录"勾选框
- 5. 用户点击"确定"：
  - 5.1 若输入框中尚有未追加的文本，先把它作为新标签追加
  - 5.2 应用把"显示在根目录"勾选状态并入展示集合后，按下述规则写入该播客的新标签集合：移除该播客旧的"对话框初始公共标签"、加入"对话框最终展示集合"
  - 5.3 写入完成后，应用发出"订阅源变更"事件，订阅页与抽屉页据此即时刷新
- 6. 用户点击"取消"：对话框关闭，不写入任何变更

## 场景二

### 场景概述

用户在"订阅页"对单个订阅卡片调出长按菜单，选择"编辑标签"

### 场景逻辑步骤

- 1. 用户进入"订阅页"
- 2. 用户对某个订阅卡片长按，弹出上下文菜单
- 3. 用户选择"编辑标签"
- 4. 应用以该订阅为参数弹出"编辑标签"对话框
- 5. 后续行为与场景一第 3 步起的"编辑标签"对话框完全一致

## 场景三

### 场景概述

用户在"订阅页"多选模式下批量编辑多个订阅的标签

### 场景逻辑步骤

- 1. 用户进入"订阅页"，长按某个订阅卡片进入多选模式
- 2. 用户选中两个或更多订阅
- 3. 用户在多选操作栏点击"编辑标签"
- 4. 应用以所选订阅的偏好列表为参数弹出"编辑标签"对话框，并在对话框顶部显示一条"公共标签"信息条
- 5. 加载时对话框展示"公共标签"（所有选中订阅共有的标签）；用户在对话框中的增删操作将作用到所选的全部订阅
- 6. 用户点击"确定"：对每个所选订阅执行相同的"移除公共旧标签 + 加入新标签集合"流程，并各自写入本地存储、各自发出"订阅源变更"事件
- 7. 用户点击"取消"：对话框关闭，不写入任何变更

## 场景四

### 场景概述

应用启动或标签变化后，"订阅页"按"标签分组"重新组织订阅展示

### 场景逻辑步骤

- 1. 用户进入"订阅页"，或停留在"订阅页"期间收到"订阅源变更"事件
- 2. 应用异步读取所有订阅及其标签，并按以下规则聚合：
  - 2.1 固定在第一位放置代表"全部订阅"的伪标签，文案显示为"全部"
  - 2.2 用户自定义标签按字母序排列
  - 2.3 当存在没有任何自定义标签的订阅时，在末尾追加代表"未分组"的伪标签，文案显示为"未分组"
- 3. 应用按以下规则展示顶部标签列表：
  - 3.1 当且仅当至少存在一个用户自定义标签时，顶部标签列表可见；否则整体隐藏，订阅列表直接展示"全部"
  - 3.2 上次离开"订阅页"时选中的标签会被恢复为当前选中态
- 4. 用户点击某个标签 chip：把下方订阅列表过滤为该标签下的订阅（"全部"对应全部订阅，"未分组"对应没有任何自定义标签的订阅）
- 5. 标签切换后，订阅列表的内容与计数立即刷新

## 场景五

### 场景概述

用户使用"抽屉式导航"布局时，标签分组在抽屉中以可折叠分组的形式展示

### 场景逻辑步骤

- 1. 用户在"全局应用设置"中关闭"底部导航"（启用抽屉式导航布局）
- 2. 用户打开侧边抽屉
- 3. 抽屉按以下规则组织订阅：
  - 3.1 任何"显示在根目录"或仅属于"全部"伪标签的订阅，平铺到抽屉顶级
  - 3.2 每个用户自定义标签作为一个可折叠分组；分组按字母序排列
  - 3.3 没有任何自定义标签的订阅归入"未分组"分组
- 4. 用户点击某分组：展开 / 折叠该分组下的订阅列表
- 5. 抽屉式导航与底部导航两种布局互斥；用户切换布局后，标签分组的展示形态随之切换，但标签数据本身不变

## 场景六

### 场景概述

"标签与分组"功能的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下入口: "播客设置页"的"标签"项（场景一）；"订阅页"订阅卡片长按菜单的"编辑标签"（场景二）；"订阅页"多选模式的"编辑标签"批量入口（场景三）
- 2. **仅作用于** 以下展示面: 底部导航布局下的"订阅页顶部标签列表"（场景四）；抽屉式导航布局下的"抽屉分组"（场景五）；"编辑标签"对话框的自动补全候选（场景一第 3.2）
- 3. **不作用于** 以下位置: "单集列表" / "单集详情页" / "播放页" / "队列页" / "收件箱页" / "收藏页" / "下载页" / "历史页" 等非订阅维度页面均不读取也不展示标签分组；"全局应用设置"主页面不展示标签数据（除"上次选中的标签"仅作 UI 状态恢复，不进入设置页 UI）
- 4. **不作用于** 以下数据范围: 单集本身没有标签；标签只属于"订阅"，不影响单集元数据

# OPML导入SPEC

用户可通过"添加播客页"或"设置 → 备份与恢复"中的 OPML 导入入口，或通过系统"打开方式 / 分享至应用"动作，把一个 OPML 文件中的若干播客批量加入订阅；导入成功后这些播客出现在订阅列表中。

## 场景一

### 场景概述

用户从"添加播客页"或"设置 → 备份与恢复"启动 OPML 导入，进入"OPML 导入页"

### 场景逻辑步骤

- 1. 用户进入"添加播客页"，点击"OPML 导入"按钮；或进入"设置 → 备份与恢复"，点击"OPML 导入"项
- 2. 应用调用系统文件选择器（不限定文件类型）让用户选择一个 OPML 文件
- 3. 若系统没有可用的文件选择器：应用提示"无法启动系统文件管理器"，导入流程不启动
- 4. 用户在系统文件选择器中选择一个文件后，应用打开"OPML 导入页"
- 5. "OPML 导入页"加载时显示进度条

## 场景二

### 场景概述

用户从系统"打开方式 / 分享"动作直接打开 OPML 文件

### 场景逻辑步骤

- 1. 用户在系统文件管理器、邮件客户端或浏览器中点击一个 OPML 文件或链接（mime 类型为 `text/x-opml` / `text/xml` / `application/xml` 之一）
- 2. 系统弹出"打开方式"，应用出现在候选列表中
- 3. 用户选择用本应用打开
- 4. 应用直接打开"OPML 导入页"并解析该文件
- 5. 该入口与场景一最终汇入同一"OPML 导入页"，后续步骤相同

## 场景三

### 场景概述

"OPML 导入页"解析 OPML 文件后展示可导入的播客列表

### 场景逻辑步骤

- 1. 应用读取所选文件的字节流
- 2. 应用按字节顺序标记探测字符编码；无标记时默认 UTF-8
- 3. 应用解析 OPML 文件：
  - 3.1 解析所有 outline 节点，提取每个节点的订阅源地址与标题
  - 3.2 跳过没有订阅源地址的节点
  - 3.3 当节点没有显式标题时，按"标题优于文本、文本兜底、最终回退为订阅源地址"的顺序确定显示标题
- 4. 解析成功后：
  - 4.1 应用以多选列表形式展示所有解析出的播客标题
  - 4.2 列表初始状态为"全部选中"
  - 4.3 顶部菜单提供"全选 / 取消全选"快速切换
- 5. 用户可：
  - 5.1 取消选中部分条目（仅导入所选条目）
  - 5.2 通过"全选 / 取消全选"切换
  - 5.3 点击"取消"按钮直接放弃导入并关闭页面

## 场景四

### 场景概述

OPML 文件解析失败或文件读取失败时的容错

### 场景逻辑步骤

- 1. 若用户进入"OPML 导入页"时没有任何文件传入：应用弹出"未选择文件"提示
- 2. 若读取文件流失败且错误与读取权限相关：应用申请存储读取权限；用户授权后自动重试解析；用户拒绝时弹出"需要读取权限"提示并允许再次申请或取消关闭页面
- 3. 若 OPML 文件格式不合法导致解析抛出异常：应用弹出"错误"对话框，标题为"OPML 读取错误"，正文附加原始错误详情；用户点击"确定"后关闭页面

## 场景五

### 场景概述

用户确认导入所选项，应用批量写入订阅并刷新

### 场景逻辑步骤

- 1. 用户在"OPML 导入页"列表中确认选择集合后点击"确认"按钮
- 2. 应用按如下分支处理：
  - 2.1 若"父级控制"已开启且"订阅需密码"已启用：先弹出父级控制密码对话框；用户输入正确后才进入下一步；用户取消则放弃导入
  - 2.2 否则直接进入下一步
- 3. 应用显示进度条，对每个被选中的播客：
  - 3.1 用其订阅源地址与标题构造一个新订阅
  - 3.2 把该订阅写入应用的本地订阅库（不写入单集级状态）
- 4. 全部写入完成后，应用立即对所有未刷新的订阅源发起一次刷新，以拉取每个新订阅的元数据与单集
- 5. 应用跳转到主界面（清空回退栈到主页），新订阅出现在"订阅页"中
- 6. 若导入过程中抛出异常：应用弹出 toast 显示异常消息

## 场景六

### 场景概述

"OPML 导入"功能的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下入口: "添加播客页"的"OPML 导入"按钮（场景一）；"设置 → 备份与恢复"的"OPML 导入"项（场景一）；系统"打开方式 / 分享"动作（场景二，仅当文件 mime 为 `text/x-opml` / `text/xml` / `application/xml` 之一）
- 2. **仅作用于** 以下数据范围: 批量"新增订阅"（写入订阅库 + 触发订阅源刷新）；不修改任何已存在的订阅
- 3. **不作用于** 以下数据范围: 单集级状态（Queue、Favorites、Playback history、Downloads、播放位置）完全不被 OPML 导入触及；OPML 导入与"数据库导入"（整体恢复应用数据库）是两个独立功能，互不影响
- 4. **不作用于** 以下位置: 单集详情页、播放页、队列页、收件箱页、收藏页、下载页、历史页等均无 OPML 导入入口

> [偏差] 需求未提及"父级控制密码门控"和"批量选择预览"；代码默认弹出多选预览列表并允许用户取消选中部分条目，且在父级控制启用时强制先验证密码。本规格如实保留这两个行为。

# OPML导出SPEC

用户可在"设置 → 备份与恢复"中把当前订阅列表导出为 OPML 文件；导出过程通过系统文件选择器让用户指定保存位置与文件名；导出不修改任何订阅数据。

## 场景一

### 场景概述

用户从"设置 → 备份与恢复"启动 OPML 导出，选择保存位置

### 场景逻辑步骤

- 1. 用户依次进入"更多 → 设置 → 备份与恢复"
- 2. 用户点击"OPML 导出"项
- 3. 应用调用系统"新建文档"选择器，让用户选择保存位置与文件名：
  - 3.1 默认文件名为 `antennapod-feeds-{当前日期戳}.opml`，用户可在选择器中修改
  - 3.2 文件类型限定为 `text/x-opml`
- 4. 用户在选择器中确认保存位置后，应用显示"请稍候"进度对话框

## 场景二

### 场景概述

系统文件选择器不可用时的降级导出路径

### 场景逻辑步骤

- 1. 用户点击"OPML 导出"项
- 2. 应用尝试调用系统"新建文档"选择器
- 3. 若系统中没有可用的文件选择器：应用以浮动提示"无法启动系统文件管理器"反馈，并回退到本地导出路径
- 4. 本地导出路径：
  - 4.1 应用在自身私有目录的 `export/` 子目录下创建以"默认文件名"命名的文件
  - 4.2 若该文件已存在则先覆盖删除
  - 4.3 写入完成后，应用通过系统内容共享机制暴露该文件
- 5. 用户确认保存后，应用显示"请稍候"进度对话框

## 场景三

### 场景概述

应用把当前订阅列表序列化为 OPML 文档并写入目标位置

### 场景逻辑步骤

- 1. 应用读取当前订阅库中的全部订阅
- 2. 应用按以下规则筛选要导出的订阅：
  - 2.1 仅导出处于"已订阅"状态的订阅
  - 2.2 跳过"已归档"状态的订阅
  - 2.3 跳过"未订阅预览"状态的订阅
- 3. 应用以 UTF-8 编码、OPML 2.0 版本写出 OPML 文档：
  - 3.1 HEAD 段写入固定标题"AntennaPod subscriptions"与以 RFC822 格式表达的创建日期
  - 3.2 BODY 段为每个被导出的订阅生成一个 `<outline>` 节点，节点包含以下属性：`TEXT` / `TITLE`（取订阅标题）、`TYPE`（订阅类型字段非空时才写入）、`XMLURL`（订阅源下载地址，始终写入）、`HTMLURL`（订阅对应的网站链接非空时才写入）
- 4. 文档写入完成，关闭输出流
- 5. 应用隐藏进度对话框，并以浮动提示反馈"导出成功"及目标文件位置

## 场景四

### 场景概述

导出过程发生异常

### 场景逻辑步骤

- 1. 在打开目标输出流、写入 OPML 文档或关闭流的任一环节发生异常时
- 2. 应用隐藏进度对话框
- 3. 应用弹出"导出失败"错误对话框，正文包含原始异常消息
- 4. 整个过程不会修改任何订阅数据；用户可在排除问题（如存储空间不足、目标位置不可写）后重新尝试导出

## 场景五

### 场景概述

"OPML 导出"功能的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下入口: "更多 → 设置 → 备份与恢复"页面的"OPML 导出"项
- 2. **仅作用于** 以下数据范围: 把当前"已订阅"状态的订阅列表导出为 OPML 文本文件；不修改任何订阅数据
- 3. **不作用于** 以下数据范围: "已归档" / "未订阅预览"状态的订阅不会被导出；单集状态（队列、收藏、播放历史、下载状态、播放位置）不会出现在 OPML 文件中
- 4. **不作用于** 以下功能: "数据库自动备份"是另一个独立功能，它把整个应用数据库备份为 `application/x-sqlite3` 文件，不是 OPML；"HTML 导出" / "收藏导出"是同页面下的兄弟项，输出不同格式且仅导出单集级数据，不在本规格范围
- 5. **不作用于** 以下位置: 添加播客页、订阅页、单集详情页、播放页等均无 OPML 导出入口；OPML 导出仅通过"设置 → 备份与恢复"触发

用户在"单集信息"分组调整单集卡的封面与时间显示

  - 3.4 用户选择正好 2 个后点击"确认"，对话框关闭，新按钮配置被持久化；下次播放时通知栏显示这 2 个按钮

# 数据持久化SPEC

应用的所有主要数据（订阅、播放队列、收藏、播放历史、下载状态）与用户设置在用户操作发生时即写入应用的私有持久化存储；应用被关闭并重新打开后，这些数据与设置按上次保存的状态恢复，不会回到错误的全空状态。

## 场景一

### 场景概述

应用持久化的数据范围与载体

### 场景逻辑步骤

- 1. 应用持久化以下"主要数据"，每类对应一种持久化载体：
  - 1.1 **订阅**：每个已订阅播客的元数据、订阅级偏好（自动下载、标签、认证、播放速度等）——存入应用私有数据库的"订阅"表
  - 1.2 **单集**：每个单集的元数据与状态（已读 / 未读 / 已播放 / 发布时间）——存入应用私有数据库的"单集"表
  - 1.3 **媒体**：每个单集对应的媒体文件信息（播放进度、上次播放时间、下载日期、文件大小）——存入应用私有数据库的"媒体"表；下载完成的媒体文件本体存入应用私有外部存储
  - 1.4 **播放队列**：用户加入队列的单集及其顺序——存入应用私有数据库的"队列"表
  - 1.5 **收藏**：用户标星的全部单集——存入应用私有数据库的"收藏"表
  - 1.6 **播放历史**：单集的播放进度与"已播放"完成时间——存入应用私有数据库的"媒体"表对应字段
  - 1.7 **下载日志**：每次下载的结果与失败原因——存入应用私有数据库的"下载日志"表
  - 1.8 **章节**：单集的章节信息——存入应用私有数据库的"章节"表
- 2. 应用持久化以下"用户设置"，载体为应用私有的偏好存储：
  - 2.1 全局外观 / 行为 / 播放 / 下载 / 同步等偏好
  - 2.2 同步账号凭据
  - 2.3 睡眠定时器最近配置
  - 2.4 使用统计与"不再提示"标记
- 3. 上述全部载体位于应用的私有存储区，操作系统保证其不被其它应用访问，且在应用进程被杀死后仍然存在

## 场景二

### 场景概述

用户做出任何会改变持久化状态的动作时，新状态立即写入持久化存储

### 场景逻辑步骤

- 1. 用户在应用中执行以下任一动作：
  - 1.1 完成一次新订阅
  - 1.2 把单集加入队列 / 移出队列 / 清空队列 / 调整队列顺序
  - 1.3 收藏 / 取消收藏某个单集
  - 1.4 播放 / 暂停 / 拖动进度 / 完成播放
  - 1.5 下载 / 取消下载 / 删除下载
  - 1.6 修改任意全局设置或播客级设置
- 2. 应用在动作完成的瞬间把新状态写入对应的持久化载体：
  - 2.1 涉及订阅 / 队列 / 收藏 / 播放状态 / 下载状态 / 下载日志的，写入应用私有数据库的对应表
  - 2.2 涉及用户设置的，写入应用私有的偏好存储
  - 2.3 涉及下载文件本体的，写入应用私有外部存储
- 3. 写入完成后，应用向当前可见的界面广播"状态变更"事件，使界面即时刷新
- 4. 写入是异步的，不阻塞用户操作；写入失败时记录错误日志

## 场景三

### 场景概述

应用关闭并重新打开后，主要数据与设置按上次保存的状态恢复

### 场景逻辑步骤

- 1. 用户通过系统正常关闭应用（按返回键退出、切换到其它应用、系统资源回收等）
- 2. 用户再次打开应用
- 3. 应用启动时执行以下恢复流程：
  - 3.1 初始化全部偏好模块，从偏好存储加载上次保存的设置到内存
  - 3.2 在首个界面显示前，从应用私有数据库读取订阅 / 队列 / 收藏 / 播放历史 / 下载状态等数据
  - 3.3 据此渲染"首页"、"订阅页"、"队列页"、"收藏页"、"播放历史"、"下载页"等界面
- 4. 恢复后的状态与用户上次操作后的状态一致：
  - 4.1 上次订阅的播客仍出现在订阅列表
  - 4.2 上次加入队列的单集仍按原顺序出现在队列
  - 4.3 上次收藏的单集仍出现在收藏
  - 4.4 上次的播放进度 / "已播放"状态仍保留
  - 4.5 上次的下载记录与下载状态仍保留
  - 4.6 上次修改的全部用户设置（主题、播放秒数、自动下载开关等）仍生效
- 5. 整个恢复过程不需要用户进行任何"显式恢复"操作

## 场景四

### 场景概述

应用升级时，已有数据按版本迁移规则保留

### 场景逻辑步骤

- 1. 用户安装应用的新版本
- 2. 应用首次启动新版本时，检测到旧版本数据库的 schema 与新版本不一致
- 3. 应用按内置的版本迁移规则，对数据库执行结构变更：
  - 3.1 增加新字段、重命名字段、调整默认值
  - 3.2 把旧字段的值按规则回填到新字段
  - 3.3 清理已失效的引用（如指向已删除单集的收藏记录）
- 4. 迁移过程中尽量保留用户已有的数据；个别 schema 差异较大时，可能丢失极少数非核心字段，但不会丢失订阅、队列、收藏、播放进度、下载状态等主要数据
- 5. 迁移完成后，应用按新 schema 继续工作

## 场景五

### 场景概述

数据库文件损坏时的异常恢复

### 场景逻辑步骤

- 1. 在极少数异常情形下（如存储硬件故障、文件系统损坏、应用进程在写入中途被强制杀死），应用私有数据库文件可能损坏
- 2. 应用检测到数据库损坏时执行异常恢复：
  - 2.1 把损坏的数据库文件复制到外部存储作为备份（文件名标识为"已损坏的数据库备份"）
  - 2.2 删除损坏的数据库
  - 2.3 应用下次启动时重建一个空的数据库
- 3. 该路径下应用会回到空状态（无订阅、无队列、无收藏、无播放历史、无下载记录），但用户设置（偏好存储）与已下载的媒体文件本体不受影响
- 4. 用户可从外部存储的备份文件尝试人工恢复，或从之前的 OPML / 数据库备份重新导入
- 5. 这是异常恢复路径，正常运行下不会触发

## 场景六

### 场景概述

"数据持久化"功能的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下数据范围: REQ 列举的全部主要数据（订阅、队列、收藏、播放历史、下载状态）+ 用户设置；写入发生在用户做出对应动作的瞬间；读取发生在应用启动与各界面进入前台时
- 2. **仅作用于** 以下载体: 应用私有 SQLite 数据库、应用私有的偏好存储、应用私有外部存储中的媒体文件
- 3. **不作用于** 以下数据范围: 内存中的瞬时 UI 状态（如滚动位置、当前展开的 chip、当前选中的标签 chip 等）不属于本规格的"主要数据"范畴，应用关闭后会丢失；个别 UI 状态（如"上次选中的标签"）虽单独持久化，但仅作 UI 状态恢复，不属于主要数据
- 4. **不作用于** 以下数据范围: 进行中的下载分片缓存不属于持久化范畴；只有下载完成后才落入持久化的"下载日志"与最终媒体文件
- 5. **不作用于** 以下数据范围: 同步队列中的"待同步事件"仅在用户配置了同步账号后才有意义；同步未配置时不写入这些事件
- 6. **不作用于** 以下情形: 应用被用户主动"清除数据"或"卸载"时，所有持久化数据被系统清除；本规格仅覆盖正常关闭 / 重启 / 升级场景

> [偏差] 需求强调"不会回到错误的全空状态"；代码在正常运行下确实不会回到全空状态，但在数据库文件损坏的异常情形下会触发"备份 + 重建空库"的恢复路径（场景五）。需求未提及"应用升级时的数据迁移"与"已下载媒体文件的文件系统持久化"，本规格如实补充。
