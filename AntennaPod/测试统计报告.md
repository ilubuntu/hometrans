# AntennaPod 迁移统计报告

## 总体评价

本次 AntennaPod 迁移完整跑完了 HomeTrans 流程，也生成了可构建、可安装的 HarmonyOS HAP，但迁移质量偏低：**最终产物主要是页面原型和本地 mock 状态演示，没有形成真实可用的播客应用闭环**。

AntennaPod 的核心不是静态页面，而是 RSS 数据、在线 HTTP 请求、播客数据库、音频下载、播放引擎、OPML、同步和设置体系。Android 应用虽然已经预置了较完整的数据，截图阶段也采到了丰富页面，但这些截图只代表“当前屏幕长什么样”，不能自动迁移背后的数据库、RSS 解析、网络请求和媒体引擎。

最终判断：**流程执行成功，但功能迁移不达标；对 AntennaPod 这类复杂在线数据型大应用，当前流程能力不足。**

## 一、项目与迁移结果

### 1.1 Android 项目概况

AntennaPod 是播客客户端，核心流程包括添加订阅、解析 RSS、刷新单集、查看订阅和详情、加入队列、下载音频、播放音频、收藏、播放历史、Inbox、OPML 导入导出、播放/下载设置等。

项目 Kotlin/Java 主源码约 **8.0 万行**。复杂度主要来自真实数据和外部能力，而不是页面数量。

### 1.2 结果摘要

完整输出见 [FINAL_REPORT.md](/Users/bb/work/hometrans/AntennaPod/output_copy/FINAL_REPORT.md)。

| 维度 | 结果 | 评价 |
|------|------|------|
| 流程 | batch UI、incremental UI、spec generate、convert pipeline 都跑完 | 流程成功 |
| UI | 采集 70 个 Android 页面状态，生成 58 个 ArkTS/TS 文件 | 页面壳较多，但不等于功能覆盖 |
| 数据 | 使用 `SampleData`、`SampleFeedFetcher`、本地 relationalStore | 样例数据持久化，不是真实数据迁移 |
| RSS/HTTP | 真实 RSS HTTP parse 被主动 defer | 核心能力缺失 |
| 下载/播放 | `LocalDownloadService`、`LocalPlaybackService` 模拟 | 无真实下载和真实 AVPlayer 播放 |
| 自测 | 14 例，9 PASS / 0 FAIL / 5 UNKNOWN | 只能说明 mock 范围内部分流程可走 |
| 代码评审 | Round 2：6 PASS / 26 PARTIAL / 8 FAIL | 大部分场景未完整实现 |

## 二、为什么截图完整但最终效果不好

### 2.1 Batch UI 只迁移“看到的画面”

`batch_ui_align` 阶段通过 ADB 抓 Android 页面截图、view tree 和 meta。AntennaPod 因为预置了数据，所以截图中 Home、Subscriptions、Feed detail、Episode detail、Player 等页面看起来比较完整。

但 batch 阶段只能识别：

- 页面上有什么文本、图片、按钮；
- 控件层级和坐标；
- 当前页面的视觉状态。

它不能从截图还原：

- RSS feed 如何请求；
- Android 数据库如何建模；
- episode 如何刷新；
- 下载文件如何落盘；
- 播放器如何解码和同步进度；
- OPML 如何导入导出；
- 在线搜索如何请求 Apple/fyyd/PodcastIndex。

所以，**截图完整只能帮助生成页面外观，不能保证功能闭环迁移。**

### 2.2 Incremental UI 只修了 5 个页面的局部视觉差异

`fix_checklist.md` 显示增量对齐只覆盖了 5 个页面：

- Home
- Subscriptions
- Feed detail
- Episode detail
- Audio player

修复项主要是字号、padding、圆角、图标尺寸、封面尺寸、BottomNav 间距、MiniPlayer 高度等。它没有覆盖 Add podcast 完整入口、Settings 分类页、OPML、Download log、Feed settings、Tags、多选、滑动操作等复杂 UX。

因此，incremental UI 能改善局部页面观感，但不能保证全应用 UX 还原。

### 2.3 Logic plan 主动 defer 了核心能力

`logic/plan.md` 明确写了：

| 证据 | 含义 |
|------|------|
| `Service interfaces ... local impls this stage, real engines deferred` | 本阶段只做接口和本地模拟 |
| `FeedFetcher -> SampleFeedFetcher` | 订阅数据来自样例数据 |
| `Real RSS HTTP parse = deferred` | 不实现真实 RSS 解析 |
| `Real media HTTP fetch = deferred` | 不实现真实媒体下载 |
| `real AVPlayer decode/render = deferred` | 不实现真实播放 |
| `OPML import/export, gpodder sync, online-provider search ... stubbed` | OPML、同步、在线搜索都是占位 |
| `Do NOT implement real RSS HTTP parsing, media HTTP download, or AVPlayer decode` | 明确禁止本阶段实现核心引擎 |

这说明功能不闭环不是偶然遗漏，而是计划阶段就把真实 RSS、HTTP、下载、播放等能力排除在本轮迁移之外。

## 三、关键证据

### 3.1 数据和引擎都是 mock/stub

`FINAL_REPORT.md` 明确写了“数据源 Mock”：RSS 拉取、媒体下载、AVPlayer 播放、OPML、gpodder、在线搜索都通过 service 接口注入本地模拟实现。

HarmonyOS 工程里也能看到对应实现：

| 文件/实现 | 说明 |
|-----------|------|
| `common/data/SampleData.ets` | 预置 feed、episode、queue |
| `SampleFeedFetcher` | 从样例数据返回 feed，不发真实 HTTP |
| `LocalDownloadService` | 模拟下载状态和进度 |
| `LocalPlaybackService` | 模拟播放状态 |
| `SettingsPage.ets` | 多个设置能力是 `not available in this build` |
| `DownloadsPage.ets` | Download log 是 toast |
| `FeedDetailPage.ets` | Feed settings / Feed info 等入口未真实实现 |
| `SearchPage.ets` | Online provider search 不可用 |

### 3.2 Review 报告显示功能大量不完整

`review-round-2/code-review-report.md`：

| 轮次 | PASS | PARTIAL | FAIL |
|------|------|---------|------|
| Round 1 | 8 | 22 | 10 |
| Round 2 | 6 | 26 | 8 |

Round 2 仍失败的主要场景：

| 场景 | 问题 |
|------|------|
| More 跳转设置页 | Settings 是扁平页面，不是分类设置中心 |
| AddPodcast 页面内容 | 缺推荐网格、本地文件夹、Apple/fyyd/PodcastIndex/OPML 等入口 |
| Download log | 页面缺失 |
| Feed settings | 页面缺失 |
| Tags | 标签编辑 UI 缺失 |
| OPML 导入/导出 | stub |
| 设置页分类 | 无分类导航、无搜索 |

这些不是边角功能，而是播客应用的重要功能面。

### 3.3 自测存在“假通过”

`TEST_REPORT_RSS.md` 指出：

- 推荐播客订阅 PASS，但订阅的是 mock 预置播客，不是真实 RSS。
- 输入任意真实 RSS URL 会失败，因为 URL 不在 `SampleData` 白名单里。
- 下载用例 PASS，但只是状态机模拟，没有真实文件。
- 播放用例 PASS，但只是进度模拟，没有真实音频。
- OPML 用例 UNKNOWN，实际仍是 stub。

所以 9 PASS / 0 FAIL 不能说明功能迁移成功，只能说明 mock 范围内部分流程能走。

## 四、时间与 Token

| 项目 | 结果 |
|------|------|
| 整体 opencode 任务 | 约 23h40m |
| opencode 总 token | 约 163.27M |

耗时和 token 都不低，但没有换来真实业务闭环，说明这类复杂应用上当前流程投入产出比不理想。

## 五、主要问题汇总

| 分类 | 问题 | 影响 |
|------|------|------|
| UI 采集能力边界 | 截图只能表达视觉状态，不能迁移业务来源 | 页面像，但功能不通 |
| UX 覆盖不足 | incremental 只对齐 5 个页面，且多是局部样式修正 | 全局 UX 还原差 |
| 真实数据缺失 | RSS/HTTP 数据源被 mock | 订阅、刷新、Inbox、Episodes 不是真实数据闭环 |
| 媒体能力缺失 | 下载和播放都是模拟 | 无真实下载、无真实音频播放 |
| 功能面缺失 | Settings、Add podcast、OPML、Feed settings、Tags、Download log 不完整 | 大量核心流程无法使用 |
| 自测可信度不足 | PASS 多在 mock 范围内，UNKNOWN 占 35.7% | 不能证明迁移成功 |

## 六、结论

AntennaPod 本轮迁移效果不好，主要不是因为 Android 数据没准备好。Android 端已经预置了数据，截图也采到了。真正原因是：

1. UI 阶段只消费截图和 view tree，迁移的是视觉状态，不是数据来源。
2. 增量 UI 只覆盖少量页面和局部样式，不覆盖完整 UX。
3. 逻辑阶段主动把 RSS、HTTP、下载、播放、OPML、在线搜索等核心能力 defer。
4. 生成的 Repository/relationalStore 只是样例数据和状态持久化，不是真实 AntennaPod 数据层。
5. 自测主要验证 mock 流程，不能发现真实业务不闭环的问题。

最终评价：**AntennaPod 本轮迁移是流程跑通，但质量不达标；当前 HomeTrans 流程更适合静态数据或本地业务较简单的应用，不适合直接迁移 RSS/新闻/播客/音视频这类依赖在线数据和平台能力的大应用。**
