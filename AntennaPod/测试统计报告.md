# AntennaPod 迁移统计报告

## 总体评价

本次 AntennaPod 迁移虽然完整跑完了 HomeTrans 流程，也生成了可构建、可安装的 HarmonyOS HAP，但从结果看，**迁移质量偏低，基本没有形成真实可用的功能闭环**。

AntennaPod 不是普通的静态页面应用，它的核心能力依赖应用内数据和在线数据：RSS 订阅、feed 解析、单集刷新、音频下载、播放引擎、OPML 导入导出、在线播客搜索、同步服务等。当前迁移产物主要完成了页面壳、部分导航、本地样例数据展示和少量状态操作；真正支撑 AntennaPod 使用的业务数据来源和引擎能力基本没有迁移。

输出证据也支持这个判断，尤其是 `logic/plan.md` 不是“遗漏实现”，而是主动把这些能力标记为本阶段不做：

- `FINAL_REPORT.md` 明确写了“数据源 Mock”：RSS 拉取、媒体下载、AVPlayer 播放、OPML、gpodder、在线搜索都通过本地模拟或接口占位实现。
- `logic/plan.md` 明确要求 `Do NOT implement real RSS HTTP parsing, media HTTP download, or AVPlayer decode`，并在 `Unknown` 中把这些列为 `Deferred engines`。
- 代码里有 `SampleData.ets`、`SampleFeedFetcher`、`LocalDownloadService`、`LocalPlaybackService`，说明数据主要来自预置样例和模拟服务。
- 多个页面仍是 toast/stub，例如 `Download log not available in this build`、`Feed settings` toast、`OPML` stub、`Online provider search is not available in this build`。
- 代码评审 Round 2 仍有 **8 FAIL / 26 PARTIAL / 6 PASS**，失败项集中在 Add podcast、Settings、Feed settings、OPML、Download log、Tags 等关键功能。

因此，本次结论应调整为：**流程执行成功，但迁移结果只达到可运行页面原型；对于 AntennaPod 这种依赖预置数据、在线拉取数据和媒体能力的大型应用，当前流程没有体现出完整迁移能力。**

## 一、转换结果对比

### 1.1 Android 项目概况

AntennaPod 是一个播客客户端应用，核心流程包括添加订阅、解析 RSS、刷新播客单集、查看订阅列表、查看单集详情、加入播放队列、下载音频、播放音频、收藏、查看播放历史、管理 Inbox、导入导出 OPML，以及配置播放和下载设置。

项目规模较大：Kotlin/Java 主源码约 **8.0 万行**。它的复杂度不只来自页面数量，更来自业务数据和外部能力：feed 数据来自网络，单集列表需要解析 RSS，播放依赖媒体引擎，下载依赖网络和文件系统，OPML 依赖文件导入导出。这类应用如果只生成 UI 和本地 mock 数据，不能认为完成了迁移。

## 二、输出证据

### 2.1 最终报告中的证据

`FINAL_REPORT.md` 里直接说明：

| 证据 | 含义 |
|------|------|
| “数据源 Mock” | 迁移产物没有接入真实 AntennaPod 数据来源 |
| “RSS 拉取、媒体下载、AVPlayer 播放、OPML、gpodder、在线搜索通过 service 接口注入本地模拟实现” | 关键外部能力没有真实实现 |
| “真实引擎延迟” | 当前版本不是完整迁移产物 |
| “OPML 导入导出、gpodder 同步、Feed 设置页、下载日志、设置中心分类页、抽屉导航应单独立项开发” | 多个整页功能缺失 |

### 2.2 logic plan 中的证据

`logic/plan.md` 明确把真实引擎排除在本轮范围外，这不是后续测试才发现的问题，而是计划阶段主动做出的范围裁剪：

| 证据 | 含义 |
|------|------|
| `Service interfaces (injectable; local impls this stage, real engines deferred)` | 本阶段只做接口和本地实现，真实引擎延后 |
| `Do NOT implement real RSS HTTP parsing` | 不实现真实 RSS 解析 |
| `Do NOT implement media HTTP download` | 不实现真实媒体下载 |
| `Do NOT implement AVPlayer decode` | 不实现真实播放引擎 |
| `SampleFeedFetcher returns sample feeds/episodes` | 添加订阅使用样例数据 |
| `LocalDownloadService simulates progress` | 下载只是模拟进度 |
| `LocalPlaybackService drives state only` | 播放只是驱动状态，不是真实音频播放 |
| `OPML import/export, gpodder sync, online-provider search ... stubbed` | OPML、同步、在线搜索都是占位 |
| `Deferred engines (stubbed, non-blocking)` | plan 将这些关键能力归为非阻塞的延后项 |

这说明逻辑阶段没有真正迁移 AntennaPod 的核心数据能力，只是做了接口、本地数据库表、样例数据和模拟服务。对于 AntennaPod 这种应用，RSS 数据库、在线 HTTP 请求、媒体下载和播放不是附加能力，而是主业务闭环本身；这些能力被主动 defer 后，迁移产物自然只能停留在演示原型。

### 2.3 代码中的证据

HarmonyOS 工程里可以看到这些实现：

| 文件/实现 | 说明 |
|-----------|------|
| `common/data/SampleData.ets` | 预置样例 feed、episode、queue 数据 |
| `common/service/SampleFeedFetcher` | 根据样例数据返回 feed，不是真实 RSS 网络解析 |
| `common/service/LocalDownloadService` | 模拟下载状态和进度 |
| `common/service/LocalPlaybackService` | 模拟播放状态 |
| `SettingsPage.ets` | 多个设置能力只是 `not available in this build` |
| `DownloadsPage.ets` | Download log 只是 toast |
| `FeedDetailPage.ets` | Feed settings、Feed info、share 等多个入口未真实实现 |
| `SearchPage.ets` | Online provider search 不可用 |

所以这里不能把“生成了 Repository 和 relationalStore”理解为“业务数据层完成”。它更准确的状态是：**有本地样例数据和状态持久化框架，但缺少真实数据接入和核心业务引擎。**

## 七、发现的问题

| 分类 | 问题 | 证据 | 影响 |
|------|------|------|------|
| 大应用适配能力不足 | 对依赖在线数据和媒体能力的应用，流程只生成页面壳和 mock 状态 | RSS、下载、播放、OPML、搜索均为 mock/stub | 无法形成真实播客应用闭环 |
| 真实数据层缺失 | 产物没有接入真实 RSS/feed 数据，只是 SampleData + 本地持久化 | `SampleData.ets`、`SampleFeedFetcher` | 订阅、刷新、Inbox、Episodes 等数据都不是来自真实业务源 |
| 功能闭环不足 | 添加订阅、下载、播放、设置、OPML 等关键链路不完整 | 8 FAIL、26 PARTIAL | App 只能演示，不能真实使用 |
| UI 采集覆盖不等于功能覆盖 | 70 个页面快照包含重复状态和滚动状态 | batch 输出 70 个 `page_*` | 页面数量不能代表迁移质量 |
| 自测可信度不足 | 9 PASS / 0 FAIL 被 UNKNOWN 和测试框架问题稀释 | 5 UNKNOWN、Round 1 全 UNKNOWN | 不能用自测结果证明质量 |
| 最终报告口径偏乐观 | FINAL_REPORT 把 mock 设计描述为非缺陷 | 同时又列出大量 deferred features | 容易高估迁移效果 |
| Benchmark 评价风险 | 如果只看是否构建成功和是否有页面，会误判为迁移成功 | HAP 可运行但核心能力缺失 | 评价体系必须加入功能闭环和真实数据能力 |

## 八、结论

**HomeTrans 能跑完整流程，也能生成一个可安装的 HarmonyOS 页面原型；但对于 AntennaPod 这种依赖应用内预置数据、在线拉取数据、媒体播放和复杂设置体系的大型应用，本轮结果没有完成真实业务迁移，基本没有形成完整功能闭环。**

这说明当前流程更适合：

- 页面较少的应用；
- 数据来源简单或本地静态的应用；
- 业务逻辑集中、外部服务依赖少的应用；
- 可以用 mock 数据演示主流程的迁移实验。

当前流程不适合直接评价为可迁移：

- RSS/新闻/播客这类依赖在线内容拉取的应用；
- 音视频播放、下载、同步等强平台能力应用；
- 大量功能依赖真实账号、网络、文件、后台任务的应用；
- 需要完整业务闭环而不是页面壳的产品级迁移。

最终评价：**AntennaPod 本轮迁移是流程跑通，但迁移质量不达标；它暴露的是 HomeTrans 当前对复杂在线数据型大应用的能力不足。**
