# Calculator 迁移统计报告

## 总体评价

本次迁移的功能还原完善度为 100%，加减乘除、百分号、正负号、清除/全清、历史记录、深色模式等全部功能均已实现且手动验证通过。表达式求值引擎与 Android 原版逻辑一致，计算结果准确。

主要不足集中在 UI 视觉还原度：Android 原应用采用新拟态（Neumorphism）设计风格，使用 Material Design 矢量图标、渐变背景按钮和高程阴影效果；HarmonyOS 迁移版将这些视觉特性简化为 Unicode emoji 图标和纯色平面按钮，视觉差异明显。根本原因是 Compose 图标库和 Surface elevation 系统在 ArkUI 中没有等价实现，增量对齐阶段识别到了这些差异但评估工作量后主动跳过。

自测阶段暴露的问题主要来自 AutoTest 坐标映射精度不足（AI 视觉对自定义组件的按钮定位偏差），导致 7 个用例中仅 1 个通过，但 hdc 手动验证 7/7 全部通过，应用本身功能无缺陷。

## 一、转换结果对比

### 1.1 Android 项目概况

SiliconeCalculator 是一个新拟态风格计算器应用，使用 Kotlin + Jetpack Compose 实现。支持加减乘除四则运算、百分号、正负号切换、清除/全清、历史记录、深色模式切换。采用 Room 数据库存储历史记录，Hilt 依赖注入，Navigation-Compose 页面导航，MVVM + Repository 分层架构。

项目规模较小：主源码约 3.5k 行 Kotlin，包含 CalculatorScreen、HistoryScreen 两个主要页面，NeuButton（新拟态按钮）、CorneredFlatButton（非对称圆角按钮）两个自定义组件，以及完整的表达式求值引擎（tokenizer + 优先级解析）。

## 四、发现的问题

| 分类 | 发现的问题 | 典型现象 | 影响范围 |
|------|------------|----------|----------|
| UI 还原 — 图标 | Android 使用 Compose Material 矢量图标（DarkMode、History），HarmonyOS 使用 Unicode emoji（☀/☾/⏱）替代 | 顶部图标视觉风格完全不同，跨设备渲染不一致 | 顶部 2 个图标按钮 |
| UI 还原 — 按钮背景 | Android 使用 `linearGradient` 渐变填充 + 装饰性渐变边框，HarmonyOS 为纯色填充无边框 | 按钮视觉风格从新拟态变为扁平化 | 全部字符按钮（约 16 个） |
| UI 还原 — 新拟态阴影 | Android 使用 Surface elevation 18dp 渲染多层阴影 + 按压动画，HarmonyOS 为简单静态阴影无按压反馈 | 按钮缺乏立体感和交互反馈 | 全部字符按钮 |
| UI 还原 — 按钮间距 | Android 按钮间距由 Grid 自动布局 + aspectRatio 控制，HarmonyOS 使用固定 margin | 间距数值接近但视觉比例有差异 | 按钮网格区域 |
| AutoTest 坐标映射 | AI 视觉对含阴影/边框的自定义组件定位精度不足 | 按钮点击坐标偏差导致操作失败 | 6/7 用例受影响；hdc 手动验证 7/7 通过 |
| 签名构建成本 | build-fixer 签名 HAP 消耗 7.59M Token | PKI 证书链生成 + base64 兼容性问题 | 占全部 Token 的 13.3% |
| `@Track` 陷阱 | `@Observed` + `@Track` getter 导致运行时崩溃 | getter 无法标记 @Track，BusinessError | Round 1 首次自测启动崩溃，修复后恢复 |

> UI 还原问题的根本原因：Compose 图标库和 Surface elevation 系统在 ArkUI 中没有等价实现，增量对齐阶段识别到了全部 5 个严重差异（🔴），但仅修复了圆角和字重等属性级差异，图标替换、渐变背景和新拟态效果因工作量评估被主动跳过。

## 附录：self-tester vs AutoTest 详细说明

### 两者是什么

| | self-tester | AutoTest |
|--|------------|----------|
| 角色 | **指挥官** — 编排测试流程 | **执行者** — 在设备上实际跑用例 |
| 运行环境 | opencode 子 Agent | 设备端测试框架 |
| 模型 | GLM-5.1 | MiniMax (api.minimaxi.com) |
| 计费 | opencode 数据库 | AutoTest batch_stdout.log |
| 是否独立 | 不知道 AutoTest 的存在 | 不知道 opencode 的存在 |

### self-tester 做什么（9 个步骤）

| 步骤 | 说明 | 仅 Round 1 |
|------|------|-----------|
| S1-S2 | 校验输入、读 app.json5 获取 bundle name | ✅ |
| S3 | 写 app-metadata.json | ✅ |
| S4 | 读 test_case.md，LLM 提取结构化 JSON | ✅ |
| S5 | 调 testcases_tool.py 生成 testcases.json | ✅ |
| T1-T4 | 校验输入、读元数据、检查设备连接、定位 AutoTest 工具 | 每轮 |
| T5 | 清理 task 目录 | 每轮 |
| T6 | 调 self_test_runner.py → 安装 HAP + 启动 AutoTest 后台进程 | 每轮 |
| **T7** | **轮询等待**：每 60 秒查一次状态，直到 COMPLETED | 每轮 |
| T8-T9 | 读结果、调 report_tool.py 生成报告 | 每轮 |

### AutoTest 做什么

每个用例的执行流程：
1. 读测试步骤描述
2. 启动 App
3. 截图 → MiniMax 理解 UI 布局
4. 执行操作（点击/输入/滑动）
5. 截图 → MiniMax 判断是否符合预期
6. 产出 PASS / FAIL / UNKNOWN
