# HomeTrans 迁移能力优缺点

## 优点

1. **形成了端到端迁移流水线**

   HomeTrans 已经形成从 Android 页面采集、资源转换、UI 迁移、需求 SPEC 生成、逻辑迁移、构建修复、代码评审到设备自测的完整流程，不是单点脚本。各阶段有明确输入输出和报告，便于追踪迁移过程。

2. **多Agent协作与可重试闭环**

   HT 把迁移拆给不同 agent 和 skill 处理，包括资源转换、UI 对齐、逻辑生成、构建修复、代码评审、自测试和失败修复等。部分阶段可以独立调用，方便补跑和重试；构建失败、评审问题和自测失败也可以进入下一轮修复，形成“生成-检查-修复-再验证”的闭环。

3. **适合本地逻辑和本地数据应用**

   当应用逻辑清晰、页面数量适中、数据来源简单时，HT 可以较完整地还原核心功能。对依赖本地 JSON、状态管理、列表、详情、收藏、筛选等能力的应用，也能完成主要页面和数据流转，适合作为中小型应用迁移实验对象。

4. **过程产物可追踪，便于人工检查和复盘**

   每个迁移环节都会产出报告或中间文件，例如 UI 对齐报告、资源转换报告、trace、SPEC、构建报告、自测报告等，方便人工检查、问题定位和结果复盘，也便于后续做 benchmark 对比。

5. **UI、资源和构建迁移能快速生成工程骨架**

   批量 UI 对齐可以根据 Android 截图和 view tree 生成 ArkUI 页面结构，资源转换可以迁移部分字符串、颜色、图片等资源，增量 UI 对齐还能修正部分字体、间距、圆角、导航和布局差异。

## 缺点

1. **流程可视化和进度跟踪不足**

   完整运行依赖 Android SDK、DevEco Studio、HarmonyOS SDK、adb、hdc、Android 模拟器、HarmonyOS 模拟器、Python/uv、homegraph、opencode/Codex/Claude skill、模型 API 和 AutoTest 配置。当前缺少统一的可视化界面和流程跟踪能力，用户不容易知道当前执行到哪一步、每一步输入输出是什么、失败原因在哪里、下一步该做什么，因此学习成本高，使用体验差。

2. **缺少鸿蒙高质量代码编写能力**

   当前流程重点是迁移界面和业务逻辑，并让 HarmonyOS 工程能够构建运行，但缺少面向 HarmonyOS 的高质量代码编写和重构能力。生成代码可能使用旧 API、`router`、V1 装饰器和较粗糙的组件拆分，整体工程质量偏低。

3. **先 UI 后逻辑对在线应用支撑不足**

   当前流程先根据截图和 view tree 迁移界面，再根据需求和 SPEC 补充逻辑。对在线应用来说，页面状态高度依赖 RSS、HTTP、数据库同步、下载、播放、账号和后台任务等真实能力，先还原 UI 容易变成静态页面，后续逻辑只能补 mock、接口或占位实现，最终功能闭环和代码质量都比较差。

4. **需求和测试用例仍高度依赖人工质量**

   HT 需要人工准备 `requirements.txt` 和 `test_case.md`。需求写得浅，SPEC 和实现就会偏浅；用例和真实 Android 行为不一致，自测结果也会失真。

5. **截图和 view tree 不能代表完整业务语义**

   UI 采集能看到页面长什么样，但不能自动理解数据从哪里来、状态如何变化、网络请求怎么触发、文件如何下载、播放器如何工作。这会导致“页面像，但功能不闭环”。

6. **自动测试稳定性不够**

   AutoTest 容易受启动应用方式、bundle name/app name 混用、图标识别、坐标点击、页面等待、模型判断影响。测试结果不能直接等同于真实功能验收。

7. **缺少有效的失败快速停止机制**

   如果应用启动失败或前置状态错误，流程仍可能继续执行大量用例，造成时间、token 和测试轮次浪费，也会让报告结论变得不干净。

8. **时间和 token 成本偏高**

   长流程会产生大量模型调用、截图分析、设备操作、自测重试和上下文读取。项目越大，成本越高，自测阶段尤其容易成为主要耗时来源。

## 总体判断

HomeTrans 适合迁移和评估本地逻辑清晰、数据来源简单、页面状态容易到达的 Android 应用。它可以显著降低从零搭建 HarmonyOS 工程和页面骨架的成本。

但对于依赖在线数据、媒体能力、复杂持久化和后台任务的大型应用，当前结果更接近“可继续开发的迁移骨架”，不能直接视为完整迁移完成。后续 benchmark 应同时评价功能闭环、UI 还原、代码质量、测试可信度、时间成本和 token 成本。
