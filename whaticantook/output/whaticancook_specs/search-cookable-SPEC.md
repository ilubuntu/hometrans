# 搜索页可做筛选SPEC

搜索页提供"Cookable"可做筛选开关，开启后结果列表只展示当前食材库已满足全部必需食材的菜谱（即可做菜谱）；可做判定随食材库变化实时刷新。

## 场景一

### 场景概述

用户开启"Cookable"，结果列表只展示可做菜谱

### 场景逻辑步骤

- 1. 用户位于搜索页，"Cookable"切换项默认为未选中态，结果列表展示全部菜谱
- 2. 用户点击"Cookable"项
- 3. "Cookable"项进入选中态（背景高亮、文字反色）
- 4. 系统对每一道菜谱，判定其"必需"食材是否全部被当前食材库满足：
  - 4.1 仅"必需"食材参与判定，"可选"食材不影响可做判定
  - 4.2 当某菜谱的全部必需食材均已被满足（即缺少的必需食材为空）时，该菜谱视为"可做"
- 5. 结果列表即时过滤，只展示可做的菜谱

## 场景二

### 场景概述

用户关闭"Cookable"，恢复展示全部菜谱

### 场景逻辑步骤

- 1. 用户位于搜索页，当前"Cookable"为选中态，结果列表只显示可做菜谱
- 2. 用户再次点击"Cookable"项
- 3. "Cookable"项退出选中态
- 4. 可做过滤取消，结果列表恢复展示全部菜谱

## 场景三

### 场景概述

当前食材库为空时开启"Cookable"，结果列表为空（无可做菜谱）

### 场景逻辑步骤

- 1. 用户清空食材库（食材库中没有任何食材）
- 2. 用户在搜索页开启"Cookable"
- 3. 由于没有任何食材能满足任何菜谱的必需食材，所有菜谱均判定为不可做
- 4. 结果列表为空，展示空态（无结果提示）

## 场景四

### 场景概述

食材库补齐某菜谱全部必需食材后，该菜谱出现在"Cookable"结果中

### 场景逻辑步骤

- 1. 用户食材库中缺少某菜谱的部分必需食材，开启"Cookable"时该菜谱不在结果中
- 2. 用户补充食材，使该菜谱的全部必需食材均被满足
- 3. 该菜谱即时判定为可做
- 4. 若"Cookable"仍为开启态，该菜谱即时出现在结果列表中

## 场景五

### 场景概述

"可做筛选"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 搜索页结果列表的可做过滤
- 2. **可做判定随食材库实时刷新**：食材库增删后，各菜谱可做状态重新判定，开启 Cookable 时结果列表随之增减
- 3. **可与文本搜索、分类筛选、排序叠加**：四者相互独立，可任意组合同时生效
- 4. **不作用于** 收藏页列表——收藏页不提供可做筛选
- 5. **"可选"食材不影响可做判定**：某菜谱含可选食材且未满足时，只要必需食材全满足，仍判定为可做

> [偏差] 需求描述未排除"无必需食材的菜谱"，实际可做判定对"无必需食材的菜谱"同样视为可做（即使食材库为空）；当前内置菜谱均含必需食材，故空食材库下开启 Cookable 结果为空，与需求验收一致。
