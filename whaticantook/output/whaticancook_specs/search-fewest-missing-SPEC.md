# 搜索「缺料最少」排序规格文档

## 需求概述
在搜索页面，用户可以按「缺料最少（Fewest missing）」排序菜谱结果。该排序方式根据用户当前拥有的食材与每个菜谱所需必需食材的对比，将缺少必需食材数量最少的菜谱排在最前面。当多个菜谱缺少数量相同时，按食材匹配比例从高到低排列。该功能帮助用户快速找到最容易完成的菜谱。

## 原子场景

### 场景1: 选择 Fewest missing 排序
- **触发条件**: 用户已进入搜索页面，食材库非空或为空均可
- **动作序列**:
  1. 用户在排序选项区域点击「Fewest missing」选项标签
- **预期行为**:
  - 「Fewest missing」标签变为选中高亮状态
  - 其他排序标签（Best match、Quickest）取消高亮
  - 菜谱列表立即按缺料最少规则重新排列
- **逻辑步骤**:
  1. 接收排序选项变更事件，更新当前排序状态为 Fewest missing
  2. 对每个菜谱计算其缺少的必需食材数量（缺失数）
  3. 对每个菜谱计算其匹配比例（已拥有必需食材数 / 必需食材总数）
  4. 按缺失数升序排列作为主排序键
  5. 对缺失数相同的菜谱，按匹配比例降序排列作为次排序键
  6. 刷新菜谱列表渲染

### 场景2: 食材库含部分食材时的排序结果
- **触发条件**: 用户已在食材库添加 carrot、green onion、garlic，排序选为 Fewest missing
- **动作序列**:
  1. 系统自动根据当前食材库重新计算排序
- **预期行为**:
  - 缺料数为 0 的菜谱（可做）排在最前
  - 缺料数为 1 的菜谱紧随其后
  - 缺料数越多的菜谱越靠后
  - 缺料数相同的菜谱中，匹配比例高的排前面
- **逻辑步骤**:
  1. 获取当前食材库列表
  2. 对每个菜谱执行食材匹配，得出缺失必需食材列表
  3. 缺失数 = 缺失列表长度
  4. 匹配比例 = 已拥有必需食材数 / 必需食材总数
  5. 主键升序排列缺失数，次键降序排列匹配比例
  6. 渲染排序后的列表

### 场景3: 空食材库下的排序行为
- **触发条件**: 食材库为空，排序选为 Fewest missing
- **动作序列**:
  1. 系统根据空食材库计算排序
- **预期行为**:
  - 所有菜谱缺失数均等于各自的必需食材总数
  - 排序退化为按匹配比例降序（所有比例均为 0，再按稳定排序）
  - 不崩溃，不报错
- **逻辑步骤**:
  1. 食材库为空时，每个菜谱的匹配比例均为 0
  2. 缺失数 = 必需食材总数
  3. 按缺失数升序排列（必需食材少的菜谱靠前）
  4. 所有比例相同时保持稳定排序

### 场景4: 切换排序选项后的实时更新
- **触发条件**: 当前排序为 Fewest missing
- **动作序列**:
  1. 用户切换到 Best match 或 Quickest
  2. 用户再切回 Fewest missing
- **预期行为**:
  - 每次切换立即重新排序并更新列表
  - 切回 Fewest missing 时恢复之前的排序结果（若食材库未变）
- **逻辑步骤**:
  1. 排序状态变更触发列表重新计算
  2. 排序逻辑为纯函数，相同输入产生相同输出

### 场景5: Fewest missing 与其他筛选组合
- **触发条件**: 用户选择了某个分类筛选（如 Dinner），且开启 Cookable 筛选，排序为 Fewest missing
- **动作序列**:
  1. 系统在筛选后的子集上应用排序
- **预期行为**:
  - 先按文本/分类/Cookable 过滤
  - 再对过滤后的结果按 Fewest missing 排序
  - 排序仅作用于已筛选的菜谱子集
- **逻辑步骤**:
  1. 应用文本搜索过滤
  2. 应用分类过滤
  3. 应用 Cookable 过滤
  4. 对过滤后结果按缺失数升序、匹配比例降序排序

## 边界场景
- **所有菜谱缺失数相同**: 退化为按匹配比例降序排列
- **菜谱无必需食材（essentialCount=0）**: 缺失数为 0，匹配比例为 1.0，排在最前
- **排序结果仅一个菜谱**: 正常渲染单项
- **排序结果为空**: 不触发排序逻辑，显示空态（参见搜索空态规格）

## 偏差说明
- 排序计算不依赖网络请求，完全基于本地数据
- 可选食材缺失不计入排序键，仅必需食材参与计算
- 排序稳定，相同输入始终产生相同顺序
