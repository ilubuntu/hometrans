# 搜索页Best match排序SPEC

搜索页排序项"Best match"按当前食材库的匹配度对结果列表从高到低排序；匹配度为某菜谱"已满足的必需食材数 / 必需食材总数"，匹配度相同者按菜谱标题字母顺序排列。

## 场景一

### 场景概述

用户选择"Best match"，结果列表按匹配度从高到低排序

### 场景逻辑步骤

- 1. 用户位于搜索页排序条
- 2. 用户点击"Best match"项（该项默认处于选中态）
- 3. "Best match"项进入选中态，其余排序项退出选中态
- 4. 系统计算每道菜谱的匹配度：已满足的必需食材数 ÷ 必需食材总数（取值 0 到 1）
- 5. 结果列表按匹配度从高到低重新排列
- 6. 每张菜谱卡片同步显示其匹配计数（状态胶囊），与排序所依据的匹配度一致

## 场景二

### 场景概述

匹配度相同时，按菜谱标题字母顺序排列

### 场景逻辑步骤

- 1. 当前生效排序为"Best match"
- 2. 当两道或多道菜谱的匹配度相同时
- 3. 这些菜谱之间按其标题的字母顺序（升序）排列
- 4. 匹配度不同者仍严格按匹配度从高到低排列

## 场景三

### 场景概述

食材库变化时，"Best match"排序结果实时刷新

### 场景逻辑步骤

- 1. 当前生效排序为"Best match"，结果列表按当前食材库的匹配度排列
- 2. 用户在食材库中增加或移除任一食材
- 3. 系统立即重新计算每道菜谱的匹配度
- 4. 结果列表按新匹配度从高到低重新排列，卡片匹配计数同步刷新

## 场景四

### 场景概述

"Best match"排序与其他筛选叠加，且只对已过滤结果重排

### 场景逻辑步骤

- 1. 用户在搜索页可能已输入搜索词、选定分类、开启"Cookable"
- 2. 系统先按搜索词、分类、可做筛选过滤出结果集合
- 3. 再对过滤后的结果集合按"Best match"匹配度从高到低重排
- 4. 排序只改变结果集合的顺序，不改变其成员（即不会因排序增减菜谱）

## 场景五

### 场景概述

"Best match"排序的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 搜索页结果列表的排序
- 2. **不作用于** 首页菜谱列表——首页列表有独立的排序，不受搜索页"Best match"影响
- 3. **可与文本搜索、分类筛选、可做筛选叠加**：排序作用于已过滤结果，四者可任意组合

> [偏差] 需求描述未明确同匹配度时的次序，实际同匹配度菜谱按标题字母升序排列，为合理的实现细节。
