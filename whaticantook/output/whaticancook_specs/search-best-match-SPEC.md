# Search Best match 排序 规格文档

## 需求概述
搜索页提供"最佳匹配"排序方式，作为默认排序。该排序依据每个菜谱相对用户食材库的"匹配比例"（已拥有的必需食材数 / 必需食材总数）从高到低排列，让"凭手头食材最接近能做"的菜谱排在前面；当多个菜谱匹配比例相同时，再按菜谱标题升序排列作为稳定的次级排序。匹配比例会随食材库变化实时重算，使排序始终反映当前食材拥有情况。

## 原子场景

### 场景1: 默认最佳匹配排序
- **触发条件**: 用户进入搜索页。
- **动作序列**: 用户查看结果列表。
- **预期行为**: 默认排序为"最佳匹配"，结果按匹配比例从高到低排列。
- **逻辑步骤**:
  1. 默认选中"最佳匹配"。
  2. 计算每个菜谱的匹配比例（已拥有必需食材数 / 必需食材总数）。
  3. 按比例降序排列。

### 场景2: 按匹配比例降序
- **触发条件**: 结果列表存在多个菜谱，各菜谱匹配比例不同。
- **动作序列**: 用户查看结果顺序。
- **预期行为**: 匹配比例高的菜谱排在前面，比例低的排在后面。
- **逻辑步骤**:
  1. 取每个菜谱的匹配比例。
  2. 按比例降序排序。

### 场景3: 比例相同按标题升序
- **触发条件**: 多个菜谱匹配比例相同。
- **动作序列**: 用户查看这些菜谱的相对顺序。
- **预期行为**: 比例相同的菜谱再按标题升序排列，保证顺序稳定可预期。
- **逻辑步骤**:
  1. 先按比例降序。
  2. 比例相同者按标题升序。

### 场景4: 无必需食材菜谱的比例处理
- **触发条件**: 某菜谱无任何必需食材（仅有可选食材）。
- **动作序列**: 该菜谱参与排序。
- **预期行为**: 该菜谱匹配比例视为最高（满分），在最佳匹配排序中靠前。
- **逻辑步骤**:
  1. 必需食材数为 0。
  2. 比例视为 1（满分）。
  3. 按满分参与降序排序。

### 场景5: 排序随食材库实时变化
- **触发条件**: 用户在食材库增删食材。
- **动作序列**: 食材库内容变化。
- **预期行为**: 各菜谱匹配比例重新计算，最佳匹配顺序随之刷新——补齐食材后相关菜谱前移，删除食材后后退。
- **逻辑步骤**:
  1. 食材库变更触发重新计算各菜谱匹配比例。
  2. 按新比例降序 + 标题升序重排。
  3. 刷新结果顺序。

### 场景6: 切换到最佳匹配排序
- **触发条件**: 当前为其它排序（最快/最少缺失）。
- **动作序列**: 用户点击"最佳匹配"标签。
- **预期行为**: 排序切换为最佳匹配，结果按匹配比例降序 + 标题升序重排；标签高亮。
- **逻辑步骤**:
  1. 更新排序为最佳匹配。
  2. 按比例降序 + 标题升序排序。
  3. 高亮标签。

### 场景7: 验证示例（carrot/green onion/garlic）
- **触发条件**: 食材库含胡萝卜、葱、蒜。
- **动作序列**: 查看最佳匹配排序结果。
- **预期行为**: 命中这些食材较多的菜谱（如鸡肉炒饭命中 3 个必需，比例较高）排在更前。
- **逻辑步骤**:
  1. 计算各菜谱比例。
  2. 鸡肉炒饭命中 3/7，比例较高。
  3. 高比例菜谱排在更前。

## 边界场景
- 全部菜谱比例相同（如食材库为空，比例均为 0）：退化为按标题升序排列。
- 无必需食材菜谱比例视为满分（1），靠前。
- 匹配比例计算遵循"已拥有必需食材数 / 必需食材总数"，与"菜谱匹配计数"规格一致。
- 最佳匹配为默认排序，且与"最快""最少缺失"互斥单选。

## 偏差说明
- 无。规格按需求描述：按匹配比例降序，再按标题排序。迁移 HarmonyOS 时应实现等价的比例计算（与"菜谱匹配计数 match-count-SPEC.md"一致）与两级排序（比例降序 + 标题升序）。
- 次级排序为标题升序（字符串序），保证比例相同时顺序稳定；这是为可预期顺序设计的稳定排序。
