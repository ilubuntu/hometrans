# 主题持久化规格文档

## 需求概述
用户在设置页面选择的主题模式应持久化存储到本地。应用重启后，应自动恢复用户上次选择的主题模式，保持主题设置的一致性。无需用户每次启动后重新设置。

## 原子场景

### 场景1: 主题选择后持久化
- **触发条件**: 用户在设置页选择新主题
- **动作序列**:
  1. 用户点击某个主题选项（如 Dark）
- **预期行为**:
  - 主题模式立即生效
  - 主题设置写入本地持久化存储
  - 存储键值：theme_mode = "DARK"
- **逻辑步骤**:
  1. 接收主题选择事件
  2. 将主题枚举名称写入本地键值存储
  3. 更新内存中的主题状态流
  4. 主题即时生效

### 场景2: 重启后恢复主题
- **触发条件**: 用户之前选择了 Dark 主题，然后关闭并重新打开应用
- **动作序列**:
  1. 用户重新启动应用
- **预期行为**:
  - 应用启动后直接以深色主题显示
  - 不闪烁、不先显示浅色再切换
  - 设置页中 Dark 选项为选中状态
- **逻辑步骤**:
  1. 应用启动时初始化设置仓库
  2. 从本地存储读取 theme_mode 值（"DARK"）
  3. 反序列化为 ThemeMode.DARK
  4. 主题状态流初始值为 DARK
  5. 应用根组件根据 DARK 设置深色配色
  6. 启动画面保持直到主题就绪

### 场景3: 首次安装默认主题
- **触发条件**: 应用首次安装，本地无主题存储记录
- **动作序列**:
  1. 用户首次打开应用
- **预期行为**:
  - 主题默认为 Match system（跟随系统）
  - 读取存储值为 null -> 使用默认值 SYSTEM
- **逻辑步骤**:
  1. 读取 theme_mode 键返回 null
  2. 反序列化 null -> 默认 ThemeMode.SYSTEM
  3. 主题状态流初始值为 SYSTEM

### 场景4: 切换主题后重启
- **触发条件**: 用户在浅色模式下运行应用
- **动作序列**:
  1. 用户切换到 Light 主题
  2. 关闭应用
  3. 重新打开应用
- **预期行为**:
  - 重启后仍为 Light 主题
  - 设置页 Light 选项高亮
- **逻辑步骤**:
  1. 切换时写入 theme_mode = "LIGHT"
  2. 重启时读取 "LIGHT" -> ThemeMode.LIGHT
  3. 应用浅色配色

### 场景5: 持久化数据容错
- **触发条件**: 本地存储中的主题值被损坏或为无效值
- **动作序列**:
  1. 应用启动读取到无效的 theme_mode 值（如 "INVALID"）
- **预期行为**:
  - 不崩溃
  - 回退到默认主题 Match system
- **逻辑步骤**:
  1. 读取 theme_mode = "INVALID"
  2. 反序列化时枚举匹配失败
  3. 匹配失败 -> 返回默认值 SYSTEM
  4. 使用 SYSTEM 主题启动

### 场景6: 主题持久化与其他设置独立
- **触发条件**: 用户修改主题和引导完成状态
- **动作序列**:
  1. 用户选择 Dark 主题
  2. 完成引导流程
- **预期行为**:
  - 主题和引导状态分别存储在不同键中
  - 互不干扰
- **逻辑步骤**:
  1. theme_mode 和 onboarding_complete 使用不同的存储键
  2. 各自独立读写

## 边界场景
- **存储值大小写**: 反序列化时大小写不敏感，"dark" 和 "DARK" 均可识别
- **清除应用数据**: 清除数据后主题存储被删除，下次启动恢复默认 SYSTEM
- **多次快速切换**: 每次切换都写入存储，最终值为最后一次操作的主题

## 偏差说明
- 主题以枚举名称字符串存储，反序列化时有容错处理
- 持久化在主题选择时同步完成，确保重启可恢复
- 默认主题为 Match system，未设置时跟随系统
