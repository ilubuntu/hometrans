# 主题持久化SPEC

用户在"设置"页面选择的主题会被持久保存；应用后续每次启动时，自动读取并生效上次保存的主题。

## 场景一

### 场景概述

用户选择某个主题后，该主题被持久保存

### 场景逻辑步骤

- 1. 用户在"设置"页面外观区块选择"Light"或"Dark"或"Match system"中的任一项
- 2. 应用将该主题对应的名字写入应用的私有配置存储
- 3. 写入立即完成；同时当前会话立即按新主题刷新配色（即时生效部分见 `theme-switch-SPEC.md`）
- 4. 该保存值在应用关闭后依然保留，不会被清除

## 场景二

### 场景概述

应用重新启动后，自动读取并生效上次保存的主题

### 场景逻辑步骤

- 1. 用户此前已选择并保存了某个主题（例如"Dark"）
- 2. 用户关闭并重新打开应用
- 3. 应用启动时从私有配置存储读取上次保存的主题值
- 4. 将读取到的主题作为本次启动后的生效主题，整应用直接按该主题呈现配色
- 5. "设置"页面外观区块中，与该生效主题一致的选项处于选中高亮状态

## 场景三

### 场景概述

从未设置过主题或保存值无法识别时，启动后回退为"Match system"

### 场景逻辑步骤

- 1. 用户从未在"设置"页面选择过任何主题，或此前保存的主题值已损坏、无法识别
- 2. 用户打开应用
- 3. 应用启动时读取主题值，发现为空或无法识别
- 4. 生效主题回退为默认值"Match system"（跟随系统深浅色模式）
- 5. "设置"页面外观区块中"Match system"选项处于选中高亮状态

## 场景四

### 场景概述

用户再次选择新主题时，覆盖旧保存值

### 场景逻辑步骤

- 1. 应用已保存了某个主题（例如"Dark"），且当前按"Dark"呈现
- 2. 用户进入"设置"页面，选择另一个主题（例如"Light"）
- 3. 应用将新主题名写入私有配置存储，覆盖此前保存的"Dark"
- 4. 当前会话立即按"Light"刷新配色
- 5. 此后再次重启应用，生效主题为最新保存的"Light"，而非"Dark"

## 场景五

### 场景概述

"主题持久化"的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下内容: "Light / Dark / Match system"三个主题选择项的跨启动保存与恢复；应用全部页面的配色方案
- 2. **不作用于** 以下内容: 菜谱数据、食材库内容、收藏列表、用户在搜索页输入的搜索词或所选分类 — 这些数据有各自独立的存储，与主题持久化互不影响

> [偏差] 需求未明确"无法识别时回退默认"的处理；代码在保存值为空或无法识别时回退为"Match system"（见场景三）。需求未明确默认主题；代码默认主题为"Match system"，与"设置"页面展示一致。均为补充说明，不与需求冲突。
> [交叉引用] 主题切换的即时生效（无需重启）见 `theme-switch-SPEC.md`；主题选项所在的"设置"页面区块结构见 `settings-overview-SPEC.md`。三份共享同一主题状态源。
