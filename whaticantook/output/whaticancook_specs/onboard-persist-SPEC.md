# 引导完成状态持久化SPEC

用户完成或跳过引导后，"引导完成"状态被持久化保存；应用在此后的每次启动中读取该状态，已完成时直接进入首页、不再展示引导。

## 场景一

### 场景概述

用户完成或跳过引导后，引导完成状态被持久化保存

### 场景逻辑步骤

- 1. 用户在引导页点击"Skip"按钮，或浏览到末页点击"Start cooking"按钮（两种入口行为等价，见 `onboarding-SPEC.md` 场景三、场景四）
- 2. 应用把"引导完成"状态写为"已完成"，并同时写入持久化存储与内存
- 3. 本次启动内，应用据此离开引导页进入首页

## 场景二

### 场景概述

已完成引导后，应用重启直接进入首页、不再展示引导

### 场景逻辑步骤

- 1. 用户此前已完成或跳过引导，"引导完成"状态已为"已完成"并保存在持久化存储中
- 2. 用户关闭应用后再次打开（冷启动）
- 3. 应用启动时从持久化存储读取"引导完成"状态，载入内存
- 4. 读取到的状态为"已完成"，应用起始页设为首页，不展示引导
- 5. 应用进入首页（Discover 首页，主标题"What can I cook"）

## 场景三

### 场景概述

从未完成引导时，应用重启仍以引导页为起始页

### 场景逻辑步骤

- 1. 持久化存储中"引导完成"状态缺省为"未完成"（应用首次安装或被清除数据后即处于此态）
- 2. 用户打开应用
- 3. 应用启动时从持久化存储读取"引导完成"状态，载入内存
- 4. 读取到的状态为"未完成"，应用起始页设为引导页，展示引导内容（展示行为见 `onboarding-SPEC.md`）

## 场景四

### 场景概述

引导完成状态的作用范围与不作用范围

### 场景逻辑步骤

- 1. **仅作用于** 以下判定：应用每次冷启动时决定起始页是引导页还是首页
- 2. **写入入口仅有** 引导页的"Skip"按钮与末页"Start cooking"按钮两处，二者行为等价；应用内其他任何页面均不能写、不能改"引导完成"状态
- 3. **不作用于** 以下情形：用户已在首页正常使用时的运行期导航——该状态仅在启动目的地判定时被消费一次，不参与运行期页面切换
