# Search Quickest 排序 规格文档

## 需求概述
搜索页提供"最快"排序方式，按菜谱的烹饪耗时从短到长排列，帮助用户优先看到能快速完成的菜谱。耗时以分钟为单位的整数衡量，取自菜谱数据；耗时相同的菜谱保持稳定的相对顺序。选择"最快"后，结果列表即时按耗时升序重排。

## 原子场景

### 场景1: 切换到最快排序
- **触发条件**: 搜索页结果列表可见。
- **动作序列**: 用户点击"最快"排序标签。
- **预期行为**: 结果按烹饪耗时从小到大排列；"最快"标签高亮。
- **逻辑步骤**:
  1. 更新排序为最快。
  2. 取每个菜谱的耗时（分钟）。
  3. 按耗时升序排序。
  4. 高亮标签。

### 场景2: 耗时短的菜谱靠前
- **触发条件**: 结果列表存在多个菜谱，各菜谱耗时不同。
- **动作序列**: 用户查看结果顺序。
- **预期行为**: 耗时短的菜谱排在前面，耗时长的排在后面。
- **逻辑步骤**:
  1. 取每个菜谱耗时。
  2. 按耗时升序排序。

### 场景3: 耗时相同保持稳定顺序
- **触发条件**: 多个菜谱耗时相同。
- **动作序列**: 用户查看这些菜谱的相对顺序。
- **预期行为**: 耗时相同的菜谱保持稳定的相对顺序，不因耗时相同而额外按其它字段重排。
- **逻辑步骤**:
  1. 按耗时升序排序。
  2. 耗时相同者保持原有相对顺序（稳定排序）。

### 场景4: 最快排序与其它条件组合
- **触发条件**: 已输入搜索词或选择分类或开启可做筛选。
- **动作序列**: 用户在此基础上选择"最快"排序。
- **预期行为**: 排序在所有筛选之后应用——先按文本/分类/可做过滤，再对过滤结果按耗时升序排列。
- **逻辑步骤**:
  1. 应用文本过滤（如有）。
  2. 应用分类过滤。
  3. 应用可做筛选（如开启）。
  4. 对结果按耗时升序排序。

### 场景5: 验证示例（5 分钟菜谱靠前）
- **触发条件**: 搜索页选择"最快"排序。
- **动作序列**: 查看结果顺序。
- **预期行为**: 5 分钟的菜谱（如 5 分钟马克杯蛋糕、蜂蜜酸奶杯、芒果酸奶饮）排在更前。
- **逻辑步骤**:
  1. 这些菜谱耗时均为 5 分钟。
  2. 按耗时升序 → 5 分钟菜谱靠前。

## 边界场景
- 全部菜谱耗时相同：顺序保持稳定（不额外重排）。
- 耗时为整数分钟；耗时格式化展示（如"5 min""1h 30m"）属详情/卡片展示，排序仅依据原始分钟数。
- "最快"为单键升序排序，无二级排序键。
- "最快"与"最佳匹配""最少缺失"互斥单选。

## 偏差说明
- 无。规格按需求描述：按烹饪耗时从小到大展示。迁移 HarmonyOS 时应实现等价的按耗时升序排序（耗时取自菜谱数据，整数分钟）。
- 耗时相同的菜谱保持稳定相对顺序（不做二级排序），这是为简洁与可预期性设计的稳定排序。
