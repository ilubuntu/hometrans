# HomeTrans 迁移系统能力评价

> 基于 Calculator（简单应用，3 个需求，7 用例）和 WhatCanICook（复杂应用，42 个需求，48 用例）两次实际迁移的完整数据。

## 优点（按优先级排序）

1. **端到端全链路自动化**：前提是人工准备好鸿蒙空模板工程、完整需求清单（requirements.txt）和测试用例清单（test_case.md），之后从 Android 页面抓取 → 资源转换 → UI 迁移 → 规格生成 → 逻辑编码 → 构建修复 → 代码审查 → 设备自测 → 缺陷修复，整条链路不需要人工干预。其中增量 UI 对齐阶段的 phone_agent 通过多模态模型驱动 adb/hdc，自动操作双端设备到达目标页面并采集截图，全程不需要手动截屏。两个项目都实现了从源码到可运行 HAP 的全自动产出。

2. **需求驱动的代码追踪与规格生成**：spec-generate 通过 GitNexus 代码索引，把人工需求（requirements.txt）映射到 Android 源码的真实入口、调用链和数据流，生成平台无关的原子场景规格。每个 SPEC 都有对应的 .trace 代码证据文件（含源码文件路径和行号），不是凭空猜测功能。

3. **多 Agent 分工 + 闭环修复**：pipeline 把迁移拆成独立 Agent（logic-context-builder 做方案、logic-coder 写代码、code-reviewer 审查、build-fixer 修编译、self-tester 跑测试），每个 Agent 上下文独立避免爆炸。审查和自测都带「发现问题 → 修复 → 重建 → 重验」循环。Calculator 审查两轮后 38/38 PASS（[报告](calculate/output/calculator_pipeline/review-round-2/code-review-report.md)），WhatCanICook 43/48 PASS（[报告](whaticantook/output/whaticancook_pipeline/review-round-1/code-review-report.md)）。

4. **构建错误自动修复**：build-fixer 能解析 ArkTS 编译错误并循环修复（最多 20 轮），两个项目都实现了 BUILD SUCCESSFUL，不需要人工介入编译问题。

5. **模块化设计 + 完整产物链路**：每个阶段可以独立调用（如单独跑资源转换、单独跑自测），方便补跑和重试。pipeline 每个阶段都输出结构化报告，pipeline-manifest.md 汇总全流程状态，人工验收时能快速定位问题。

---

## 缺点（按优先级排序）

1. **环境依赖重、工具学习成本高、配置步骤多**：需要 DevEco Studio、Android SDK、adb、hdc、Python 3.10+、uv、多模态模型 API、GitNexus 等，任一缺失即失败。配置分散在环境变量和 ~/.hometrans/config.json 两处，Skill 流程涉及 7 个阶段、8 种子 Agent、多种工具脚本，新用户上手门槛高。且 phone_agent 和 AutoTest 独立调用多模态 API，不经过 opencode，Token 消耗不在主数据库里，实际总消耗被低估。

2. **工程质量产出偏低**：迁移产物大量使用废弃 API，页面导航用 router 而非 Navigation，状态管理用 V1 装饰器（@Observed + @Track）而非 V2，公共组件抽取有限。两个项目都存在这些问题，pipeline 没有工程质量提升阶段。

3. **UI 视觉还原度差，缺乏专项转换能力**：Compose 库生成的矢量图标、渐变背景、新拟态阴影等视觉效果无法转换，全部被降级为 emoji 和纯色平面。这些不是增量对齐阶段偷懒，而是当前模型能力解决不了 — 这些图标和效果在 Android 端是 Compose 库代码生成的，不是图片文件，转换工具无法直接搬过来。增量对齐虽然发现了这些差异并标记为严重，但没有对应的专项 Skill 来处理，只能跳过。需要专门制定「Compose 视觉资源 → HarmonyOS 资源」的转换能力（如从 Material Icons 下载 SVG 导入 HarmonyOS 工程）。

4. **流程和工具优化不足，浪费大量 Token 和时间**：
   - **缺少 fail-fast**：WhatCanICook Round 1 App 启动就崩，仍把 48 个用例全跑完，白费约 4 小时。应首个用例启动失败即中止。
   - **AutoTest 定位不准**：靠 AI 截图识别按钮位置，对带阴影和边框的自定义组件经常点偏，Calculator 7 个用例只通过 1 个，hdc 精确坐标验证 7/7 全通过，自测通过率不能真实反映应用质量。
   - **签名构建不必要**：pipeline Skill 要求 `--signed` 构建签名 HAP，但模拟器上 unsigned 就能装。Calculator 遵循了 Skill 要求，签名构建白费 35 分钟和 7.59M Token；WhatCanICook 的主 Agent 自行跳过了签名直接构建 unsigned，反而省了这笔开销。Skill 设计本身有问题（模拟器不需要签名），且不同 Agent 执行严格度不同导致结果不一致。
   - **长会话上下文膨胀**：self-tester 单个 Agent 会话跑 4 小时、200+ 轮对话，实际新内容（输入+输出+推理）约 1.06M Token，但 opencode 框架每轮对话会把完整历史记录重发给模型，导致 cache_read 达 43.9M（占 self-tester 总量的 97.7%）。虽然 cache_read 有计费折扣，但原始 Token 统计量被严重放大，影响成本估算的准确性。详见 [WhatCanICook Token 统计](whaticantook/测试统计报告.md#三token-统计)。

---

## 总结

HomeTrans 的核心价值在于把 Android → HarmonyOS 迁移拆解成了可自动执行的流水线，前提是人工准备好鸿蒙空模板工程、完整需求清单和测试用例，之后从源码到可运行 HAP 全程不需要人工写代码。功能迁移能力较强，Calculator 实现了 100% 功能覆盖（7/7 手动验证通过），WhatCanICook 覆盖了 42 个需求中的主要功能和数据链路。

不足方面，工程质量产出偏低是当前最大的短板，大量使用废弃 API 和 V1 装饰器，导航用 router 而非 Navigation，公共组件抽取有限。UI 视觉还原也有差距，Compose 库生成的矢量图标和视觉效果无法转换，导致产物「能用但不好看」。流程和工具层面存在签名构建不必要、长会话上下文膨胀、缺少 fail-fast 等设计缺陷，浪费了 Token 和时间。AutoTest 对自定义组件的定位精度不足，自测通过率不能真实反映应用质量。环境依赖重、上手门槛高也是推广的障碍。

---

*评价基于 Calculator（56.86M Token，5h 13m）和 WhatCanICook（127.30M Token，12h 53m）两次完整迁移数据*
